Why not use the same token for every app?

A token is labeled for one destination. A StaffMail token says aud: staffmail, so PayrollPro must reject it. Separate tokens stop one compromised app from opening every other app, allow different permissions and expiration times, and let each app know the proof was created specifically for it. CloudGate can issue the next token silently because it still remembers Maya’s original login.

Exactly who sends what to whom?

1 · App → browser“Go to CloudGate and sign in.”
2 · Browser → CloudGateThe browser follows the redirect.
3 · CloudGate → browser → appA short-lived, one-time authorization code returns.
4 · App backend → CloudGateThe backend exchanges that code at CloudGate’s token endpoint.
5 · CloudGate → app backendCloudGate creates and returns the signed ID token—and an access token when requested.

The browser usually carries the code, not the actual production token. The app backend receives the token directly from CloudGate over HTTPS, verifies it, and then gives the browser an app-session cookie.

Why doesn’t Maya log in again for the next app?

CloudGate also created its own secure session cookie after the first login. When another app redirects Maya to CloudGate, the browser sends that cookie only to CloudGate. CloudGate recognizes the still-valid session and can mint a new token for the new app without showing a password screen.

This means one login does not mean one reusable token. StaffMail and PayrollPro normally receive different tokens with different aud (audience) claims. The reusable memory is the IdP session; the proofs sent to apps stay narrow and short-lived.

Lab 2 · Tap each colored segment

What is inside?

A JWT is readable, signed data—not an encrypted password.

02

JWT means JSON Web Token

Pronounced “jot,” a JWT is a compact way to send claims—small facts such as who authenticated, which app the token is for, and when it expires—together with proof that those facts were not changed.

1 · HeaderInstructions: which signing algorithm and public key should the app use?
2 · PayloadClaims: who issued it, who the user is, the audience, and the expiration time.
3 · SignatureTamper evidence: proof that the trusted issuer signed the exact header and payload.

Important: a JWT is a format, like a standardized envelope—not a guarantee that the contents are trustworthy. Its contents are encoded, not encrypted, so anyone holding it can usually read them. The signature prevents undetected changes; HTTPS protects the token while it travels. An app trusts the claims only after checking the signature, issuer, audience (aud), and expiration (exp). ID tokens are commonly JWTs; access tokens may be JWTs or opaque strings.

PayrollPro ID token

Keep this:Claims become trustworthy only after the signature, issuer, time, and audience all pass validation.
ID token, access token, app session—what is the difference?

ID token: tells the client application who authenticated and facts about that authentication. In OpenID Connect it is intended for the client named by aud.

Access token: authorizes a client to call an API. Its audience is the resource server/API, and it may contain scopes such as payroll:read. An API validates it; a client should not treat it as a general-purpose identity profile.

App session: after validating the sign-in response, a traditional web app usually creates its own secure session cookie. The browser then uses that cookie on normal page requests; it does not need to replay the ID token forever.

What does “verify the signature” actually mean?

CloudGate hashes the exact encoded header and payload, then uses its private key to produce a signature. PayrollPro obtains CloudGate’s public keys from its published JWKS endpoint and selects the key named by kid.

PayrollPro verifies that the signature matches those exact bytes. If an attacker changes role, sub, aud, or exp, the bytes change and verification fails. The public key can check a signature but cannot create a new valid one.

Real libraries must also restrict acceptable algorithms, trusted issuers, audiences, and clock skew. “Decode the JWT and read the claims” is not validation.

Exact issuance, use, renewal, and revocation rules

ID token issuance: Google issues it as part of an OpenID Connect authentication response or when the application backend exchanges an authorization code. The application validates it to learn which Google subject authenticated and confirms signature, issuer, audience, expiry, and nonce. It is not the credential for arbitrary Google API calls.

Access token issuance: Google’s authorization server issues it after the user or policy authorizes particular OAuth scopes. The application sends it to the Google API named by the token’s permissions. The API needs it; the user-facing website does not use it as a replacement for its own session cookie.

Refresh token issuance: for a web-server application it is optional and normally requires offline access. The application stores it securely on its backend and sends it only to Google’s token endpoint to obtain replacement access tokens. It can stop working after user revocation, long inactivity, some password changes, token-count limits, time-limited consent, testing-mode limits, or administrator policy.

Session issuance: Google creates its account session when the user authenticates to Google. Your application creates a separate application session after it validates the Google response. Either session can outlive an individual ID or access token, and either can be ended independently.

Local JWT revocation: a self-contained JWT can be validated without calling its issuer, so its issuer cannot pull back every copy instantly. Short expiry bounds that risk. Systems needing faster shutdown add introspection, denylists, token-version checks, security-event notifications, or coordinated app-session revocation.

One-screen reference: the complete mental model

IdP: authenticates the person, writes claims, signs tokens with its private key, publishes public keys, and controls whether new tokens can be issued.

Browser: follows redirects and carries protocol messages. It can read a JWT and could try to alter it, so apps never trust it as the authority.

App: validates structure, trusted issuer, allowed algorithm and signature, time claims, nonce/state where applicable, and the exact audience. Only then does it use claims and create an app session.

Security boundary: passwords stay at the IdP; signatures reveal tampering; audience limits where a token is accepted; expiry limits replay; TLS and secure browser storage protect tokens in transit and at rest.
