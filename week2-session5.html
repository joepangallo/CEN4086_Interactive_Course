1 From DAS to the Cloud: How Storage Got Here

Four eras, one direction: away from the single machine

Each era moves storage farther away from one physical computer. What changes is who can reach the data and what the storage system sends across the network.

One machine

DAS · Direct-attached storage

A disk is inside or directly cabled to one computer or server. That machine reads and writes raw blocks.

Best at: simple, inexpensive, low-latency storage.

Limit: access, capacity, and failure are tied to that one machine.

Shared files

NAS · Network-attached storage

A dedicated appliance shares folders over the local network using file protocols such as SMB or NFS.

Best at: office shared drives and files used by many people.

Limit: it is still usually an appliance in one building.

Shared blocks

SAN · Storage area network

A dedicated high-speed network sends raw block volumes to servers using Fibre Channel or iSCSI. Each server treats its volume like a local disk.

Best at: high-performance databases and data-center workloads.

Limit: specialized hardware and expertise make it expensive.

Storage service

Cloud storage · 2006 onward

Storage becomes an on-demand service reached through APIs. The provider manages hardware, capacity, replication, and durability.

Best at: scaling without purchasing storage arrays; pay for what you use.

Limit: connectivity, service dependency, and transfer or retrieval fees still matter.

NAS and SAN are not the same. NAS sends complete files: “give me /invoices/march.pdf.” A SAN sends numbered blocks: “give me sectors 4096–8191.” Cloud platforms now offer both models, plus object storage.

2 Block vs File vs Object — and Why Data Survives

The three shapes of cloud storage

Choose the storage shape by asking: what does the workload need to read and write?

🧱 Block storage

Raw numbered blocks presented to one VM like a disk. The VM adds its own filesystem.

read blocks 4096–8191

Use for: databases, VM boot disks, frequent small updates.

Examples: AWS EBS · Azure Managed Disks

📁 File storage

A ready-made shared filesystem with folders, paths, and permissions that many clients can mount.

open /marketing/logo.png

Use for: shared documents, home directories, legacy drive-letter apps.

Examples: AWS EFS · Azure Files

🪣 Object storage

Whole blobs stored with metadata and a unique key, reached through HTTP APIs.

GET covers/2026/book.jpg

Use for: images, video, backups, logs, static websites.

Examples: S3 · Azure Blob · Google Cloud Storage

What is a raw block volume?

Think: virtual blank diskThe volume is divided into numbered, fixed-size chunks called blocks.
What storage understandsCommands such as “read block 4096.” It does not understand filenames or folders.
What the operating system addsIt partitions and formats the volume with NTFS, ext4, or another filesystem, creating files and directories.
Why databases use itThey can update one small block in place instead of replacing an entire stored object.
“Raw” does not mean unencrypted. It means the storage service supplies addressable blocks without imposing its own file or folder structure.
An S3 bucket is not a normal folder. A key such as covers/2026/book.jpg only contains slash characters that look like folders. Object storage reads or replaces the whole object; it cannot efficiently rewrite one small database block. That is why databases normally use block storage.

Replication & durability: the eleven nines

Cloud object stores protect data by keeping replicas on separate devices and often in separate availability zones.

1 · WriteStore the object
2 · ReplicateCreate copies in other locations
3 · VerifyContinuously check integrity
4 · RepairReplace a damaged copy
DurabilityWill the data still exist? “Eleven nines” describes an extremely low probability of losing an object.
AvailabilityCan you reach the data right now? Durable data can be temporarily unavailable during an outage.
ReplicationKeeps current copies alive after hardware failure—but copies deletions and ransomware too.
Backup / versioningPreserves older states so you can rewind after a mistake or attack.
Midterm rule: durability ≠ availability, and replication ≠ backup. Multiple live copies protect against failed hardware; version history protects against bad changes.

3 Tiers, Backup & Compliance

Hot, cool, archive: paying for access, not just space

Storage price depends on two things: how much data you keep and how often you retrieve it.

🔥 Hot

Highest storage price, immediate access, little or no retrieval charge.

Choose when: data is used daily or hourly.

Examples: active databases, current website assets.

🌤️ Cool

Lower storage price, but retrieval charges and minimum-storage rules may apply.

Choose when: data is used monthly.

Examples: older projects, infrequent reports.

🧊 Archive

Very low storage price, slower restore times, and high retrieval charges.

Choose when: data is rarely opened but must be retained.

Examples: compliance records, long-term history.

TodayHot: active data
After 30 daysMove to cool
After 180 daysMove to archive
After 7 yearsDelete if policy allows
“Archive everything” is not a savings strategy. Frequently restored archive data can cost more than hot storage—and users may wait hours for every restore. Match the tier to the access pattern.

Cloud backup and the 3-2-1 rule

Replication keeps a service running. Backup lets you return to an earlier, clean state.

3copies of important data
2different media or services
1offsite—and ideally immutable or offline
Retail POS example: (1) live database, (2) nightly versioned backup in cloud object storage, and (3) weekly export in an immutable archive bucket with object lock.
  • Use versioning: rewind to before deletion or ransomware.
  • Test restores: a backup is only proven when you successfully restore it.
  • Separate access: stolen production credentials should not be able to erase every backup.

Industry-specific concerns: storage with a lawyer attached

Classification comes first. Different data sets can require different locations, retention periods, and protections.

Which rules apply?PCI-DSS: payment cards · HIPAA: health records · FERPA: student records.
Where may it live?Data-residency rules may restrict the country or cloud region.
How long must it remain?Retention schedules and legal holds prevent early deletion.
How is it protected?Encryption, customer-managed keys, object lock, and WORM immutability.
Architecture rule: classify each data set, then choose its storage type, tier, region, retention period, and access controls. Lab C turns that decision into a register.

🧠 AI Concept of the Day · Embeddings, Vector Databases & RAG — storage that understands meaning

AI concepts track: every session teaches one real AI concept through that day's cloud objective. Today, while differentiating storage solutions for Objective 6, you learn the storage type AI quietly added to the catalog — and RAG, the retrieval pattern that lets a chatbot answer from documents it was never trained on.

Traditional search matches words. AI search can match meaning.

1 · Embedding

A model converts text, images, or audio into a list of numbers—coordinates representing meaning.

Similar meanings produce nearby coordinates.

“Return policy” can be near “refunds and exchanges” even without shared words.

2 · Vector database

Stores embeddings and quickly finds the nearest matches among millions of vectors.

Optimized for: “What is closest in meaning?”

Examples: Pinecone · Weaviate · pgvector · Chroma

3 · RAG

Retrieval-Augmented Generation retrieves relevant source text before the model writes an answer.

Use for: changing policies, inventory, documents, and facts requiring citations.

The model answers from retrieved context instead of guessing from memory.

Original documentsPDFs, product pages, and images remain in cheap, durable object storage.
Meaning indexEmbeddings live in the vector database and point back to the original source objects.

How RAG answers a question

1 · EmbedConvert the question into a vector
2 · RetrieveFind the closest document chunks
3 · GroundAdd those chunks to the prompt
4 · GenerateAnswer from the retrieved evidence
Quillmark Books example: the recommendation bot can use today’s inventory and store policies because those sources are retrieved at question time. Update a document, re-index it, and the bot can use the new information without retraining the model.
Uploading documents does not make the model memorize them. RAG changes the stored documents and vector index; the model’s trained parameters remain unchanged.
Use RAG for changing knowledgeInventory, policies, prices, recent memos, and facts that should be cited or permission-checked.
Use fine-tuning for lasting behaviorSkills, response style, domain vocabulary, and patterns the model should apply by default.

Q1 · The full server

Dr. Okafor's dental practice stores 14 years of patient imaging on disks inside the front-office server — classic direct-attached storage. The drives are now 91% full, the imaging software crawls, and when the server rebooted for updates last Tuesday, no one in the building could open a single X-ray for 40 minutes. Her IT contractor proposes "just buying a second server with its own disks for the new images." What fundamental DAS limitation does that proposal fail to fix?

Answer: B. ✔ The 40-minute X-ray blackout is the tell: with DAS, data's fate is welded to one machine's fate. A second server doubles capacity but also doubles the islands — now staff must know which box holds which year, each box is still a single point of failure, and each grows only as large as its own chassis. The evolutionary fix is shared storage (NAS/SAN) or cloud storage, which decouple data from any one machine. ✘ A is false — locally attached disks are typically the fastest option (that's why cloud VMs still use local disks for scratch data); speed isn't DAS's flaw, isolation is. ✘ C — no law forbids DAS for medical images; regulations govern protection and retention, not cable type. ✘ D — DAS backs up fine; the problem is it usually depends on someone remembering (Session 1's Rosa had a USB "backup").

Q2 · Thirty accountants, one drive letter

Marisol runs IT for a 30-person accounting firm in Fort Myers. Staff currently email spreadsheets to each other; version chaos is constant. She wants one central place where every employee's computer sees the same shared folders with permissions — "the S: drive experience" — served from a device on the office network she can back up centrally. She is NOT running databases and has no data-center budget. Which technology matches her need?

Answer: D. ✔ Shared folders, many simultaneous users, permissions, central backup, modest budget — that is the NAS use case verbatim: a file-serving appliance on the LAN speaking file-level protocols. (Her likely next step, a cloud file service, is NAS's descendant — same file abstraction, different building.) ✘ A — a SAN serves blocks over a dedicated fabric at enterprise cost; it's built for database servers and hypervisors, not for thirty accountants who need "the S: drive," and block volumes aren't safely shareable between ordinary workstations anyway. ✘ B doubles down on the email-attachment disease: thirty private islands, zero shared truth, thirty backup chores. ✘ C repeats the SAN category error at retail scale — raw blocks have no shared-folder semantics or permissions; each workstation would just see an unformatted disk.

Q3 · The gallery goes viral

StreetSnap, a photo-sharing startup, stores user photos as files on one big cloud file share. At 2 million photos the listing operations are slowing; at their growth rate they'll hit 80 million photos within two years. Photos are uploaded once, never edited, and served to browsers over HTTP millions of times a day. CTO Amara asks where the photos should live long-term. What should you recommend, and why?

Answer: A. ✔ Uploaded once, never edited in place, read over HTTP at massive scale — that's the object-storage profile line by line. A flat keyspace has no directory tree to lock or rebalance, which is exactly why it scales to billions of objects, and per-GB pricing plus eleven-nines durability fits a startup's budget and paranoia. ✘ B — block volumes attach to one server at a time, cap out at volume-size limits, and make the app do all the serving work; you'd be rebuilding object storage badly on top of raw disks. ✘ C mistakes the word "file" for the workload: hierarchical filesystems slow precisely on enormous flat collections (their 2-million-photo symptom is the early warning). ✘ D — RAM is a cache, thousands of times pricier per GB and gone on reboot; caches sit in front of durable storage, never instead of it.

Q4 · The sluggish checkout

Copperleaf Market's new cloud POS backend commits every sale to a PostgreSQL database that rewrites thousands of small pieces of its data files every minute. A junior architect, eager to save money after this morning's lecture, proposes hosting the database's live data files directly in object storage: "It's 23× cheaper per GB than the disks we're renting!" Why must you overrule him?

Answer: C. ✔ A live database engine constantly seeks and rewrites small regions inside big open files, demanding sub-millisecond block-level I/O — precisely what block volumes (the cloud's "SAN as a service") provide. Object storage offers no in-place edits: changing one byte means re-uploading the whole object over HTTP, so the "cheaper" plan would make every sale take seconds and corrupt consistency guarantees besides. Right tool, right shape: block for the live DB, object for its backups. ✘ A is backwards — object stores' multi-AZ replication typically makes them the durability champion; durability was never the issue. ✘ B — PCI-DSS regulates protecting cardholder data, not banning cloud databases; thousands of banks run them. ✘ D — object stores handle multi-terabyte objects; invented limits don't win design reviews.

Q5 · Selling the nines

During the client pitch rehearsal, Beacon Bicycles' skeptical owner reads the slide aloud: "Product images stored with 99.999999999% durability." He squints: "Eleven nines. Does that mean the website never goes down?" Consultant Jocelyn has one sentence to answer correctly. Which is it?

Answer: B. ✔ Durability and availability are different promises with different math: durability says the bytes survive (achieved by replicating across devices and availability zones and continuously re-verifying checksums); availability says you can reach them this second (typically quoted around 99.9x% — a real but temporary outage risk). A provider outage can make perfectly durable data briefly unreachable — annoying, not fatal. Jocelyn's owner needs exactly that distinction before he repeats the slide to his bank. ✘ A confuses the two concepts — eleven nines describes loss probability, not uptime, and no one sells eleven-nines availability. ✘ C — encryption is a security control, orthogonal to durability; "eleven ways" is numerology. ✘ D — the estimate comes from published replication design and device failure statistics; "can't be estimated" is a dodge, and consultants who dodge get replaced.

Q6 · The CFO's frozen reports

TrendMart's CFO, Bram, read that archive storage costs "$1 a terabyte" and ordered ALL 60 TB of company data — including the sales-reporting datasets analysts query every Monday — moved to the archive tier. Two months later he's furious: the monthly storage line shrank, but the total storage bill nearly doubled, and analysts wait hours each Monday for datasets to "thaw." What did Bram miss about storage tiers?

Answer: D. ✔ This is the tier trap from Segment 3 and your Lab A break-even exercise: archive's rock-bottom per-GB price is subsidized by retrieval fees and restore latency. Data touched every Monday belongs in hot (or at coolest, cool); a lifecycle policy should demote only what genuinely goes untouched — Bram paid retrieval on the same "cold" data 8+ times. The fix: classify by access pattern first, then tier — never the reverse. ✘ A — archive tiers are ordinary commercial products with no government restriction. ✘ B — archive tiers have minimum storage durations (early-deletion fees), the opposite of silent deletion; nothing is discarded. ✘ C — the ~$1/TB-month figure is standard published pricing, not a promotion; the price was real, and so were the fees printed beside it that nobody read.

Q7 · The backup that wasn't

Ransomware hits Coastal Realty at 6:14 AM Friday, encrypting every file on the office server. Office manager Denise stays calm — "we're fine, everything syncs to our cloud drive." She opens the cloud drive: every file there is encrypted too. The sync client dutifully uploaded each encrypted file within seconds of the attack. Total loss? Not quite — what SINGLE capability, had it been in place, would rescue Coastal Realty this morning, and why didn't sync provide it?

Answer: A. ✔ Sync's job is to make all copies identical fast — which on attack day means industrializing the damage. Versioned backup (or a backup service with point-in-time restore) keeps immutable history, so Denise rolls the whole share back to 6:13 AM and the ransom note becomes a war story. This is the replication-vs-backup distinction from Segment 2, now with payroll on the line; it's also why 3-2-1 wants one copy offline or immutable. ✘ B — RAID protects against disk hardware failure; it faithfully stores encrypted garbage with excellent redundancy. ✘ C — more replicas means more perfect copies of the attack; replication ≠ backup, in bold, on the midterm. ✘ D — tiering changes price and latency, not history; an encrypted file in archive is just cheaper encrypted garbage that takes hours to retrieve.

Q8 · Seven years, untouchable

Two Rivers Coffee's accountant, Mr. Whitaker, drops a requirement on you, the consultant: financial records must be kept for seven years, must be producible for auditors within a business day or two, are opened at most twice a year — and, after a franchise lawsuit he'd rather not discuss, must be provably unalterable, even by whoever holds the admin password. Which storage design meets ALL of it?

Answer: C. ✔ Map each requirement to a feature: 7-year retention → lifecycle/retention policy; twice-a-year access with day-scale tolerance → archive tier (hours-long restores are fine, and ~$1/TB-month keeps Whitaker smiling); provable unalterability → object lock/WORM, which even a compromised administrator credential cannot override until the retention clock expires. This is compliance storage as a checklist, and it drops straight into the storage half of Milestone 2. ✘ A pays hot prices ~23× over for files opened twice a year and — worse — a writable shared folder fails the "unalterable" requirement outright. ✘ B — a sticky note is not a control; NAS files are one bad click or one disgruntled admin from silent edits, and the lawsuit taught Whitaker exactly that. ✘ D is destruction of records with a retention obligation — a legal catastrophe, not a storage strategy.

Q9 · The Monday-morning oracle 🧠 AI concept

Cielo Insurance publishes a brand-new claims policy to its document portal Monday at 9:00 AM. At 9:20, an adjuster asks the company's AI assistant about hail-damage deductibles — and gets the NEW policy, quoted correctly, with a link to the source document. CTO Rios is baffled: "Training a model takes weeks of GPUs and a fortune, and we changed nothing this weekend. How can it know a document that's twenty minutes old?" What's the correct explanation?

Answer: D. ✔ Retrieval-Augmented Generation separates knowledge from the model: the parameters stayed frozen all weekend — what changed was the index. Embedding one document takes seconds and pennies, and every question triggers a fresh nearest-neighbor search, so "twenty minutes old" is no harder than "twenty months old." The citation link is the tell: grounded answers can point at their source. ✘ A — nightly retraining of a large model per customer is economically absurd (Session 1's training-vs-inference math); no vendor does it. ✘ B is today's misconception verbatim: uploads change storage, never parameters. ✘ C — guessing from stale training data would produce the OLD deductible, fluently (a hallucination risk), and it cannot explain a working link to a document that didn't exist last week.

Q10 · Two stores for one bot 🧠 AI concept

Architect Lena is designing the storage layer for Quillmark Books' recommendation bot. Two very different data sets: (1) the originals — 40,000 cover images and 9,000 publisher PDFs, written once and served to browsers; (2) an embedding vector for every book description, hit with a nearest-neighbor search on every recommendation request, all day. A junior dev shrugs: "Put everything in one system — simpler." Applying Objective 6 — differentiate storage solutions — what should Lena build?

Answer: A. ✔ Session 5's whole skill, applied to an AI workload: match each data set's access pattern to the store built for it. Write-once/read-many blobs over HTTP is object storage's exact profile (Q3's StreetSnap logic); millions of "what's closest in meaning?" queries is the one job vector databases exist for. Shared keys stitch them together — find neighbors in the index, fetch originals from the bucket. ✘ B misuses a specialized index as a blob store: vector DBs price and optimize for vectors, and gigabytes of images bloat both the index and the invoice. ✘ C chains both data sets to one machine — Session 5's DAS island reborn — while paying low-latency block prices for cold PDFs. ✘ D confuses size with temperature: archive trades hours of retrieval latency for pennies, and a search index consulted on every request is the hottest data in the system.
