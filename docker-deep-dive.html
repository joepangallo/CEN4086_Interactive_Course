Lab 1 · The mental model first

One recipe, many cakes

Before any command: what an image is, what a container is, and why they are not the same thing.

01

Here is the oldest problem in software: a program works on your computer, but when your friend tries it, it breaks. Different computer, different setup, different missing pieces. Docker fixes most of that drift with one idea: don't hand over just the program — package it with its user-space runtime, libraries, and default configuration. That package behaves consistently on compatible laptops and cloud servers because the moving parts travel together. The host still supplies a compatible kernel and CPU, which the platform experiment below makes precise.

📦 ImageThe recipe: a frozen master copy of your app and everything it needs. It never changes and it never runs. You make it once and share it everywhere.
🚢 ContainerA cake baked from that recipe: a live, running copy with its own files, its own network, its own little world. Bake one or bake fifty — all from the same recipe.
🗄️ RegistryThe cookbook library: a website that stores images so anyone can download them. Docker Hub is the big free one. docker pull = borrow a recipe; docker push = share yours.
The one sentence to memorize: an image is the frozen recipe; a container is a running copy of it. Deleting a container never hurts the image — eating a cake doesn't hurt the recipe.

Where this sits in what you already know

A virtual machine virtualizes hardware. Each VM boots and maintains a guest operating system. That stronger boundary costs extra disk, memory, and startup work.
A container isolates processes. It shares the host kernel, so there is no guest OS per app to boot or duplicate. Containers are typically much smaller and start much faster.

So what IS a container, really?

Here is the honest answer, no magic: a container is just a normal program running on your computer — with invisible walls built around it. It is not a tiny computer hiding inside your computer (that's what a virtual machine is). When a container starts, the operating system builds three walls around the program:

🗂️ Its own filesThe program starts with only the image plus its own writable layer. Host folders, volumes, devices, and sockets appear only when you explicitly mount or grant them.
🌐 Its own network doorwayIt gets a private doorway to the network. Door 80 inside the box is NOT your computer's door 80 — Lab 5 is all about connecting them.
📋 Its own process listIt sees only processes in its process namespace — often one main process, sometimes helpers — not every process running on the host.

From the inside, it feels like having a whole computer to yourself. From the outside, it is just one more program in your computer's list. The wall-builder is the kernel — the boss program at the heart of every operating system. That one fact decides the next question:

What exactly is "Docker" when I install it?

Two programs, working as a team. Who you talk to: the docker command — the messenger. Who does the work: the Docker daemon, a helper that runs in the background and does the real jobs — downloading images, building the walls, starting containers. Where they live: on Linux the daemon runs right on the machine; on Mac and Windows it lives inside the tiny hidden Linux helper you met above. When: every command you type in Lab 2 is the messenger handing one request to the daemon. Why it matters: Docker images follow a shared open standard (called OCI), so an image Docker builds is not locked to Docker — other tools and every big cloud can run it too.

Get it for free: install Docker Desktop (free for personal and education use), or use Play with Docker — a browser Linux machine with Docker pre-installed (a Docker account is required). The CLI patterns in this guide transfer to both.

What actually happens during docker run -d -p 8080:80 --name web nginx?

1 · Check: do we have the nginx image? If not, download it automatically. 2 · Create: make a brand-new container — a fresh writable box stacked on top of the read-only image, with its own walls. 3 · Wire: -p 8080:80 connects your computer's door 8080 to the box's door 80. 4 · Start: the image's default program (for nginx, the web server) starts up inside the box. 5 · Step away: because of -d, Docker prints the new container's ID and gives your prompt back — the box keeps running in the background until you stop it.

Next step · distribute it

You built the image. How does someone else use it?

Right now myapp:1.0 exists only inside your Docker Engine. The normal handoff is to upload the image to a registry such as Docker Hub and send the other person its image name plus a run command. They do not need your source code or Dockerfile because they receive the already-built image.

1 · You pushUpload the built image under a registry address and version.
2 · The registry storesDocker Hub or another registry holds the image layers.
3 · They pull and runDocker downloads the image and creates their own container from it.
Normal path · internet or team registry

Push it to Docker Hub

First create a Docker Hub repository named myapp and choose whether it is public or private. Replace YOUR_DOCKER_ID below with your Docker Hub username.

On your computer:

# Sign in, give the local image its registry address, then upload it.
docker login
docker tag myapp:1.0 YOUR_DOCKER_ID/myapp:1.0
docker push YOUR_DOCKER_ID/myapp:1.0

On the other person's computer:

docker pull YOUR_DOCKER_ID/myapp:1.0
docker run --rm -p 3000:3000 YOUR_DOCKER_ID/myapp:1.0

They open http://localhost:3000. Pressing Ctrl+C stops the foreground container, and --rm removes that used container while keeping the downloaded image.

docker run can pull a missing image automatically; spelling out pull makes the handoff visible. For a private repository, grant that person's Docker account access; they run docker login before pulling. Never send them your password or access token.

Offline path · no registry

Send one image archive

When the machines cannot reach a registry, save the image and its tag into a tar archive, then transfer that file with your approved file-sharing method.

On your computer:

docker save -o myapp-1.0.tar myapp:1.0

On the other person's computer:

docker load -i myapp-1.0.tar
docker run --rm -p 3000:3000 myapp:1.0

Use docker save for an image. docker export is a different tool: it flattens a container's filesystem and does not preserve the image's normal metadata and history for this handoff.

What you actually send: one copyable image reference, one complete run command, and “open http://localhost:3000.” Also list required environment-variable names, volume mounts, and configuration files — but send secret values through a secret manager, never inside the image or README.
Keep this:build creates the image on one machine; push publishes it; pull retrieves it; run creates the recipient's container. Prefer a meaningful version such as :1.0 over relying only on :latest.
Will the same image run on their computer?

Usually, if Docker is installed and the image matches their operating-system and CPU platform. One common surprise is building a Linux ARM image on an Apple Silicon Mac and handing it to an Intel Linux server. Emulation may help, but the distributable answer is a multi-platform image whose one registry name contains both variants:

docker buildx build \
  --platform linux/amd64,linux/arm64 \
  -t YOUR_DOCKER_ID/myapp:1.0 \
  --push .

Docker chooses the matching variant when the recipient pulls it. A version tag is convenient but can be moved to different bytes later; for an audit or deployment that must use the exact pushed image, share the sha256: digest printed by docker push and pull YOUR_DOCKER_ID/myapp@sha256:DIGEST.

Official references: tag and push, save an image, load an image, and build for multiple platforms.

RUN vs CMD — the two most-confused instructions

RUN happens while the image is being MADE. It runs a command right then (like installing libraries) and freezes the result into a layer, so that piece is saved inside the image forever.

CMD happens LATER, when a container starts. It doesn't run during the build at all — it just writes down the program every future container should start with. Run fifty containers and each one starts that program fresh.

Build & tag it yourself: in a folder with these files you would type docker build -t myapp:1.0 . — the -t names (tags) the image, and the final . says "the recipe and files are in this folder." Then docker run -d -p 3000:3000 myapp:1.0 starts it, exactly like Lab 2's nginx.

Lab 4 · Build your own, for real

Hello world from scratch — three languages

The exact steps to build your own container on your own machine with Docker Desktop, then the famous trick that shrinks it by 99%.

04

Everything before this ran boxes other people made, or simulated the building. Now the real thing: you write a program, you write the recipe, you build the image, you run the container — on your own machine. The recipe is the same in every programming language on Earth:

1 · Make a folder, write the programA completely ordinary program — nothing about it is Docker-specific. The program always comes first; the box comes second.
2 · Write the Dockerfile next to itA file named exactly Dockerfile — capital D, no extension — in the same folder. Three to six lines, and you already know every instruction in them.
3 · Build, then rundocker build -t NAME . freezes your image, docker run --rm NAME bakes a container from it (--rm throws the used container away the moment it finishes). Same two commands, every language, forever.
Only ONE thing truly changes between languages: does the program need a build-time preparation step (a RUN line), or can it run as-is? Python: no — Python reads your source fresh every start. C++: yes — it must be compiled into a machine program first. TypeScript: yes — it must be translated into JavaScript first. Watch that one difference as you click through the three tabs.

Need Docker itself? Use Docker Desktop, or open Play with Docker when you cannot install software. Lab 1 explains what Desktop adds on Mac and Windows.

Step 1 · Make the folder and the program. Open your terminal (or Docker Desktop's >_ Terminal button) and make a fresh folder:

mkdir docker-hello-python
cd docker-hello-python

Create a file named hello.py with one line in it:

# hello.py — the whole program.
print("Hello, World! from my own Python container")

Step 2 · Write the Dockerfile. In the same folder, a file named exactly Dockerfile:

# Start from a tiny Linux that already has Python installed —
# you never install Python yourself; you start from a box that has it.
FROM python:3.14-alpine

# Pick the folder inside the box to work in (creates it too).
WORKDIR /app

# Copy your program into the box.
COPY hello.py .

# What every container from this image runs the moment it starts.
CMD ["python", "hello.py"]

Notice what is missing: there is no RUN line. Python is an interpreted language — the python program reads your source code fresh every time it starts, so there is nothing to prepare during the build. Four lines and done. This is why Python is the gentlest first build.

Step 3 · Build it.

docker build -t my-python-hello .

Read it out loud: "follow the Dockerfile in this folder (.) and name the frozen result my-python-hello." The first build downloads the python:3.14-alpine base image — the current Python feature line in a small ready-made starting box; the exact download size varies by CPU architecture. After that it is cached.

Step 4 · Run it.

docker run --rm my-python-hello

You'll see Hello, World! from my own Python container — printed by your program, from inside your container, built from your recipe. The new flag: --rm means "throw the container away the moment it finishes" — perfect for say-one-thing-and-quit programs, and it keeps your docker ps -a list tidy. Check Docker Desktop's Images tab: my-python-hello is on your shelf.

Step 1 · Make the folder and the program.

mkdir docker-hello-cpp
cd docker-hello-cpp

Create hello.cpp — ordinary C++, nothing Docker-flavored about it:

// hello.cpp — the whole program.
#include <iostream>

int main() {
    std::cout << "Hello, World! from my own C++ container\n";
    return 0;
}

Step 2 · Write the Dockerfile. Here's the difference from Python: your computer cannot run hello.cpp at all — C++ must be compiled into a machine program first. Compiling is a one-time, build-time job, and you know which instruction owns build-time jobs: RUN.

# Start from a Linux that ships with the g++ compiler already installed.
FROM gcc

# Pick the folder inside the box to work in.
WORKDIR /app

# Copy your program into the box.
COPY hello.cpp .

# BUILD-TIME job: compile hello.cpp into a runnable program named "hello".
# Runs ONCE, during docker build — the finished program is frozen into a layer.
RUN g++ hello.cpp -o hello

# START-TIME: what every future container runs. The ./ means
# "the hello program right here in this folder."
CMD ["./hello"]

Step 3 · Build it.

docker build -t my-cpp-hello .

Two things to expect. First, this download is big — the gcc base image is over a gigabyte, because a whole compiler workshop is heavy (remember that feeling; the shrink trick below is the cure). Second, watch the RUN g++ step scroll by — that's your program being compiled inside the build. A typo in hello.cpp fails right there with a normal g++ error, and no image gets made: a broken program can never become a "finished" image.

Step 4 · Run it.

docker run --rm my-cpp-hello

Hello, World! from my own C++ container — then it stops cleanly (the 0 in Exited (0) means "no errors"), because a container lives exactly as long as its main program. Now peek at the Images tab and note the size: about 2 GB, for a two-word program. Hold that thought for the shrink trick below.

Step 1 · Make the folder and the program.

mkdir docker-hello-ts
cd docker-hello-ts

Create hello.ts:

// hello.ts — the ": string" part is TypeScript's addition:
// a type label that plain JavaScript doesn't have.
const message: string = "Hello, World! from my own TypeScript container";
console.log(message);

Step 2 · Write the Dockerfile. TypeScript is the interesting middle case: it IS a compiled language, like C++ — but it compiles into another language (plain JavaScript) instead of machine code, because Node can only run JavaScript. So this Dockerfile needs RUN lines too — one to install the translator, one to translate:

# Start from a tiny Linux that already has Node.js installed.
FROM node:24-alpine

# Pick the folder inside the box to work in.
WORKDIR /app

# BUILD-TIME job 1: install the TypeScript translator (tsc) into the box.
# It sits BEFORE the COPY on purpose — Lab 3's cache lesson: this slow step
# doesn't depend on your source file, so editing hello.ts never redoes it.
RUN npm install -g typescript

# Copy your program into the box.
COPY hello.ts .

# BUILD-TIME job 2: translate hello.ts → hello.js (plain JavaScript).
RUN tsc hello.ts

# START-TIME: Node runs the TRANSLATED file — hello.js, not hello.ts.
CMD ["node", "hello.js"]

Step 3 · Build it.

docker build -t my-ts-hello .

Watch the two RUN steps go by: installing the translator, then translating. Each becomes its own frozen layer — and the caching lesson from Lab 3 applies: edit hello.ts and rebuild, and the slow install layer is reused instantly.

Step 4 · Run it.

docker run --rm my-ts-hello

Hello, World! from my own TypeScript container. One detail worth noticing: the type label (: string) is gone by the time the container runs — types are checked during translation and then removed. The box runs hello.js; hello.ts is just along for the ride. (Which means the translator and the .ts file could be left out of the shipped box entirely… which is exactly the next lesson.)

The shrink trick: multi-stage builds

Do the C++ tab for real and you'll own a strange object: a 2.14 GB image carrying a two-word program. Why so huge? Because the box you ship still contains the entire compiler workshop — g++ and all its tools — which you needed for exactly one moment: the RUN g++ line, during the build. The finished, compiled hello never uses the compiler again. You baked a cake, then shipped the whole kitchen along with it.

A multi-stage build is one Dockerfile with two FROM lines — two boxes instead of one:

# ============ Stage 1: the workshop (temporary) ============
# "AS builder" gives this stage a name so stage 2 can point at it.
FROM gcc AS builder
WORKDIR /app
COPY hello.cpp .
# -static packs every helper the program needs INSIDE the program,
# so the finished file can live completely alone.
RUN g++ -static hello.cpp -o hello

# ============ Stage 2: the shipping box (what you keep) ============
# "scratch" is a special base: a COMPLETELY EMPTY box. Zero bytes inside.
FROM scratch
# The magic line: reach back into the workshop, take ONLY the finished program.
COPY --from=builder /app/hello /hello
CMD ["/hello"]

When the build ends, the workshop is left out of the final image — only stage 2 is tagged for shipping. Same build command (docker build -t my-cpp-hello-small .), same run command, same output. The numbers below are one measured build; exact sizes vary with base-image updates and CPU architecture, but the order-of-magnitude difference is the lesson:

Why does an empty box even work? This is Lab 1 paying off: the isolation and kernel come from the host. Ordinary base images provide a user-space filesystem — shells, libraries, certificates, and utilities — not a kernel. A fully static program can run from scratch; an app that needs shared libraries, CA certificates, or timezone data must copy them or use a small runtime base.

Keep this:The workshop stage builds; the shipping stage keeps only runtime requirements. A dramatic size reduction speeds pulls and removes unused tools and libraries from the attack surface. It does not make the app invulnerable — the shipped program and shared kernel still matter. The same pattern works for TypeScript: compile in one stage, then ship the JavaScript and runtime dependencies.

Which base image should I pick? A cheat sheet

python:3.14-alpine / node:24-alpine — tiny Linux with the language pre-installed. A practical starting point for these dependency-free demos, which need their interpreter in the shipping box forever. For real dependencies, confirm Alpine's musl-based environment is supported; a slim Debian variant is often the less surprising choice. Node 24 is the current LTS line used here.

gcc — the full C++ workshop. Great as a builder stage; too heavy to ship. Use it on the top floor of a multi-stage build.

debian:stable-slim — a smaller general-purpose user-space for shipping dynamically linked programs without the full compiler toolchain. It is often the gentle middle choice when scratch is too bare.

scratch (0 bytes) — the completely empty base. It needs a fully self-contained program plus any runtime data the program expects. Forgetting -static is the classic C++ gotcha: the build succeeds, but the container can die with a confusing "file not found" because its dynamic loader is absent.

Read the flag left-to-right as "outside:inside". -p 8080:80 = your computer's door 8080 leads to the box's door 80. The inside number is decided by the program in the box (nginx: 80; the Lab 3 app: 3000). The outside number is yours to pick.

Docker is read-only and writable

The phrase “Docker images are read-only” is true, but it leaves out the most important runtime detail. When Docker creates a container, it stacks a thin writable container layer on top of the image. The program sees one combined filesystem even though the pieces underneath have different owners and deletion rules.

Bottom · shared📦 Read-only image layersFROM, COPY, and RUN froze these files during the build. Containers can read them but never alter the image itself.
Top · per container✏️ Writable container layerBy default, new files and runtime changes land here. Stop preserves it; removing the container deletes it.
Plugged into a path🔌 Volume, bind, or tmpfs mountAt a mounted path such as /app/data, writes bypass the container layer and follow the mount's own lifecycle.
Copy-on-write in plain English: if a process changes an image-provided file, Docker records the changed version in that container's writable layer. If it “deletes” an image file, Docker records a hide marker there. The original image layer remains untouched, so a fresh container sees the original file again.

“Delete” can mean three different things

docker stop is not deletionThe same container still exists. docker start brings back its writable layer and files.
docker rm deletes one containerIts writable layer disappears. The image and named volumes remain; bind-mounted host files remain.
Deleting a mounted file deletes real dataInside a writable volume or bind mount, rm changes that storage directly. A new container cannot resurrect it.

Inspect before deleting

Persistence is not a backup. Before deleting a volume that matters, make and test a backup outside that volume.

Commands with different blast radii

docker rm web                 # container layer
docker rm -v web              # + its anonymous volumes
docker volume rm dbdata       # one named volume
docker volume prune           # unused anonymous volumes
docker volume prune --all     # + unused named volumes
docker compose down           # containers; volumes stay
docker compose down --volumes # + project volumes

Do not copy the cleanup commands blindly. Docker refuses to remove a volume still in use, but “unused” does not mean “unimportant.”

Want the container itself to reject writes?

That is an explicit runtime choice, not Docker's default. --read-only makes the container's root filesystem read-only; then grant only the writable locations the app truly needs. Mounts are read-write by default, so mark a bind or volume readonly separately when it should also reject writes.

# Read-only root, disposable writable /tmp.
docker run --read-only --tmpfs /tmp myapp:1.0

# Read-only root, but persistent app data may be written here.
docker run --read-only \
  --mount type=volume,src=appdata,dst=/app/data \
  myapp:1.0

# A configuration folder the container may read but not change.
docker run \
  --mount type=bind,src=./config,dst=/app/config,readonly \
  myapp:1.0

Ports + data checkpoint

A database listens on 5432 in its container, should be reached at 15432 on your laptop, and must keep its data. Which shape is right?

Named volumes vs bind mounts

Named volume (-v notes:/app/data): a storage bin Docker owns and looks after in its own space on your computer. The best default for databases and app data — it just works, no folder paths to get wrong.

Bind mount (-v your-folder:/usr/share/nginx/html): plugs a specific host path straight into the container. Developers use it for live source code and configuration: edit a file, refresh, no rebuild. It is host-path dependent and writable by default; add :ro when the container only needs to read. Mounting over a non-empty container directory temporarily hides the image's original contents at that path.

tmpfs mount (--tmpfs /run/cache): keeps temporary data in host memory rather than the container layer or disk. It disappears when the container stops and can suit disposable caches or short-lived sensitive material — never anything that must survive.

Running is not the same as ready

Docker tracks the container's main process. A HEALTHCHECK adds a second signal: can the application actually do its job? Select each state and learn what Docker has — and has not — proved.

Keep this:running proves a process exists. healthy proves your chosen test is passing. Docker Engine records an unhealthy result, but a health check alone does not repair or restart the app — an operator or orchestrator needs a policy for that.

Build-context rule: if Docker never receives a secret, COPY cannot accidentally preserve it in a layer. Still use BuildKit secret mounts when a build step itself needs a credential.

The five-command troubleshooting flight plan

1 · docker ps -a — Does the container exist? Is it running, restarting, or exited? Read the status and port columns. 2 · docker logs --tail 100 web — What did PID 1 say before or during failure? Add -f to follow live output. 3 · docker inspect web — What configuration did Docker actually apply: command, environment, mounts, network, health history, and exit code? 4 · docker stats — Is CPU, memory, network, or I/O pressure the problem? 5 · docker exec web ... — Only after the container is running, test from inside its network and filesystem view.

The order matters: first establish lifecycle, then read the app's own evidence, then inspect configuration and resources. Jumping straight into a shell can hide the original failure and does not work on a stopped container or a minimal image with no shell.

Running one container is one command. Real apps are teams: a web service, a database, perhaps a cache or worker — each in its own container, each needing consistent configuration. Docker Compose records that application model in compose.yaml. A service can be built from your Dockerfile or pulled from a registry; Compose creates a private network, volumes, health dependencies, and explicitly granted secrets around them.

# compose.yaml — the application model
services:
  web:               # service 1: our Node app
    build: .          # use the Dockerfile here
    ports:
      - "3000:3000"  # the only host-facing doorway
    environment:
      DATABASE_HOST: db  # service-name DNS, not localhost
      DATABASE_NAME: app
      DATABASE_USER: app
      DATABASE_PASSWORD_FILE: /run/secrets/db_password # app reads it
    secrets:
      - db_password
    depends_on:
      db:
        condition: service_healthy  # wait for readiness

  db:                # service 2: Postgres
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: app
      POSTGRES_USER: app
      POSTGRES_PASSWORD_FILE: /run/secrets/db_password
    volumes:
      - dbdata:/var/lib/postgresql/data
    secrets:
      - db_password
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U app -d app"]
      interval: 5s
      timeout: 3s
      retries: 10

volumes:
  dbdata:            # Docker-managed persistent storage

secrets:
  db_password:
    file: ./secrets/db_password.txt
Validate before startingdocker compose config parses the model, resolves variables, and shows the configuration Docker will use. Catch YAML and interpolation mistakes before startup noise hides them.
One command updocker compose up -d --build builds the web image, pulls Postgres, creates the volume and default network, starts db, waits for its health check, then creates web.
Names replace changing IPsThe web app connects to db:5432. Inside web, localhost means web itself — not your laptop and not Postgres.
Secrets are explicitly grantedOnly services listing db_password receive the file at /run/secrets/db_password. Keep secrets/ out of version control.
Down is intentionally conservativedocker compose down removes containers and the network but keeps dbdata. down -v also deletes declared volumes and their data.

What Compose does not do for you

It does not make every app ready-aware: short depends_on: [db] waits only until db is running, not until it accepts connections. The long form plus a health check closes that race. It does not back up volumes: persistence survives container replacement, but deletion, corruption, or disk failure still requires a backup plan. It does not turn one machine into a cluster: Compose is excellent for development, tests, demos, and many single-host deployments; multi-host scheduling and automatic rescheduling are orchestration concerns. It does not make a secret safe after you commit the source file: runtime mounting limits exposure to granted services, but you still protect and rotate the source credential.

Glossary — the 12 words to keep straight

Image: immutable filesystem layers plus startup metadata. Container: an isolated process and its writable layer, created from an image. Registry: a service that stores and distributes images. Repository: a named collection of image versions in a registry. Tag: a human-friendly, mutable label such as 24-alpine. Digest: a content-derived, immutable identifier beginning with sha256:.

Build context: the files sent to the builder; . in docker build . selects the current directory. Layer: a cached filesystem or metadata change produced by a Dockerfile instruction. Volume: Docker-managed persistent storage outside a container's writable layer. Bind mount: a direct mapping of a host path into a container. Service: Compose's definition of a container role. Health check: a repeated command whose exit code says whether the app-level probe passes.
