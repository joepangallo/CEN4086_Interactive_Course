MCP from zero — what you are actually building

1 · The definition

MCP stands for Model Context Protocol. It is an open standard for connecting an AI application to outside data and operations through one common message format. A useful analogy is USB-C: the protocol defines the plug and conversation, while the device on either side supplies the actual capability. The official architecture overview uses the same host–client–server roles defined below.

Without a standard, every AI client needs custom glue for every order system, document store, calendar, or database. With MCP, a server publishes what it offers in a discoverable form and an MCP-capable client can list and use those capabilities at runtime.

2 · The five roles in this homework

RolePlain-English jobIn this homework
HumanAsks for something and approves consequential actions.You, clicking and entering arguments in the Inspector.
HostThe AI application that manages the user experience, model, permissions, and connected servers.A production chat/IDE would be the host. The Inspector supplies only a developer test UI here.
MCP clientMaintains one protocol connection to a server, discovers capabilities, sends requests, and receives results.The Inspector's client connection.
MCP serverPublishes named capabilities with contracts, validates calls, executes code, and returns structured results.Your server.mjs process.
Backing systemsHold the real data or perform the real work behind the server.policies.json, orders.json, and tickets.json.
Critical distinction: a host, client, server, and model are not synonyms. The model may suggest a tool call; the client carries it; the server decides whether the requested operation is valid and performs it under the server's permissions.

3 · The three things an MCP server can publish

PrimitiveWhat it meansStarter exampleWho normally initiates it?

business://policies is an identifier understood by this server; it is not a public website URL. Reading it asks the MCP server to return the resource content.

4 · What happens when the Inspector connects

  1. npm run inspect starts the Inspector and launches node server.mjs as a child process.
  2. The client and server perform initialization/capability negotiation: each side identifies itself and agrees on supported protocol features.
  3. The client sends discovery requests equivalent to “list your resources,” “list your tools,” and “list your prompts.”
  4. The server returns names, descriptions, URIs, and schemas. This is why the Inspector can build a form for each tool without custom UI code.
  5. When you click Run, the client sends a structured tool call. The SDK validates its shape, your server checks business rules, the backing file changes only if allowed, and the server returns a structured result.
Inspector/client  ── initialize + list ──▶  server.mjs
Inspector/client  ◀── names + schemas ───  server.mjs
Inspector/client  ── tools/call(args) ──▶  schema → business rule → file
Inspector/client  ◀── result or error ────  server.mjs

The messages use JSON-RPC semantics. In HW5 they travel through stdio—the server's standard input/output streams—because everything is local. Remote MCP servers usually use Streamable HTTP. The transport changes how messages travel, not what a resource or tool means.

5 · What MCP does not provide

MCP does not supply an LLM, embeddings, a vector database, a user account system, authorization policy, cloud hosting, or automatic trust. Your code and host still own those decisions. The Inspector contains no model: it lets you test discovery and calls manually so every result is deterministic and free.

Understanding checkpoint: before setup, make sure you can finish these sentences aloud: “The server is…,” “the client is…,” “a resource differs from a tool because…,” “stdio is…,” and “MCP does not replace RAG because….” These ideas are assessed in Parts 1, 4, and 5.

How MCP follows RAG without replacing it

HW3/HW4 built a RAG pipeline: embed the question, retrieve the nearest policy, then answer from that evidence. MCP solves a different layer of the system.

RAG knowledge path
question → embed → retrieve policy chunk → give evidence to model → cited answer

MCP integration path
host/client → discover resource or tool → structured request → server → result

Together in production
MCP resource → host's RAG pipeline → retrieved evidence → model answer
MCP tool     → live system / approved action              → model reports result

In this homework, business://policies returns the policy collection; the Inspector simply displays it. It does not run embeddings or select a chunk. A production host could feed that resource into the RAG pipeline you learned in HW3, or the server could later publish a separate search_policies(question) tool. We are deliberately keeping the existing RAG idea and isolating the new lesson: standardized discovery and safe action.

Why MCP is the next step

Customer needRight primitiveWhy
“What is your damaged-order policy?”RAG over an MCP resourceRead changing source text, retrieve the relevant part, cite it; no side effect.
“Where is ORD-1002?”Read-only MCP toolThe answer is live application state, not prose in a document.
“Create a case for ORD-1002.”Write MCP tool + approvalThe operation changes state, so typed arguments, authorization, validation, confirmation, and logging matter.
“Handle this case using our normal steps.”MCP promptA user-selected, reusable workflow tells a client how to combine resources and tools.

The control split is intentional: the host/application chooses what resources enter context, the model may propose a tool call, and a human must remain able to approve or deny consequential calls. Read the official MCP server concepts if those roles blur together.

Part 2 — Finish the write tool's server-side boundary

Goal: implement exactly one state-changing function while keeping every denial side-effect free.

get_order_status and get_ticket already work. The starter deliberately leaves createSupportTicket() unfinished in support-store.mjs. Implement the five requirements in its TODO comment:

  1. No explicit confirmation → return CONFIRMATION_REQUIRED; write nothing.
  2. Unknown order—even a well-formed ORD-9999—→ return ORDER_NOT_FOUND; write nothing.
  3. Approved, known order → append exactly one TKT-XXXXXXXX ticket with status: "open" and an ISO timestamp.
  4. Append one JSON audit event containing the action, IDs, and timestamp—but not the issue text.
  5. Return the created ticket. Keep the schema and business checks; do not weaken the tests.
required decision order
1. confirmed is not exactly true?  → deny; stop
2. order does not exist?           → deny; stop
3. construct one ticket            → write tickets.json
4. construct sanitized audit event → append audit.jsonl
5. return the created ticket       → ok: true

Do not write first and validate later. A denied call must leave both files byte-for-byte unchanged.

Part 5 — The one-page design memo

Goal: explain when to use grounded knowledge, read tools, and approved write tools—and what MCP standardizes between them.

Write a one-page memo for the business owner:

  1. Classify six requests of your own: two answered from grounded policy documents, two needing read tools, and two proposing writes. Justify each classification.
  2. Name the three enforcement layers you observed—schema validation, server business rules, and host/human approval—and one failure each layer catches that the others do not.
  3. Explain what MCP standardizes and what it does not supply: model, RAG index, authentication policy, database, or automatic trust.
  4. Recommend which tools could be pre-approved and which always need a confirmation dialog. Use consequence, reversibility, and data sensitivity—not “AI is smart”—as your criteria.
Part 5 is done when: the memo has all four labeled sections, contains six classified examples, and explicitly says that the Inspector ran no model or inference.
