Part 1 · Trace a real SSO login

The concept — federation: one organization (the identity provider) checks who you are; another (the service provider — the app you want) accepts that answer because the two organizations agreed to trust each other before you ever showed up.
  1. Pick a site you actually use that has a Sign in with Google / GitHub / Apple / Microsoft button.
  2. Sign in once, slowly, and watch the address bar. Write down each domain you see, in order. You will usually see two different organizations' domains — that is federation happening. If the provider redirects you back too quickly to notice its page, it may have reused an existing signed-in session; record that observation instead of assuming no federation occurred.
  3. In hw1/answers.docx, write the login as a numbered list of about six steps: click the button → redirect to the provider → provider authenticates you → redirect back to the app → app verifies the result → app starts its own session. For each step, name who is acting: your browser, the app, or the login provider. A numbered list is all that's required — do not draw a diagram.
  4. Some steps (token verification) are invisible in the address bar. Include them anyway and mark them (inferred).
  5. Label three things in your write-up: the identity provider (who checked your password), the service provider (who trusted the answer), and the trust relationship between them.
  6. Finish with 3–5 sentences: why does the app believe the provider's answer, and what would the app have to build, store, and protect if that button didn't exist?
⚠️ Record domain names only (like github.com). Never copy full URLs, query strings, codes, cookies, or tokens into your repo — a URL can contain a short-lived secret.
Hand in: the numbered trace with actors, the three labels, and the trust explanation — in hw1/answers.docx.

Part 2 · Inside the token

The concept — tokens: the trust in Part 1 ultimately reaches the app in a signed message from the identity provider: "this is who the user is, valid until this time." In a common authorization-code flow, your browser carries a short-lived login response back to the app, then the app's server exchanges that response for tokens directly. The signature makes an identity token trustworthy; the expiry limits the damage if it is stolen.
  1. Work through every tab of the Inside the SSO Token interactive: anatomy, tamper lab, lifetimes, kill switch.
  2. Answer these three questions in hw1/answers.docx, one short paragraph each:
    • Roles: for the login you traced in Part 1 — who mints the token, who carries the browser-visible login response, and who verifies the token? If the token exchange was invisible, label your answer as inferred.
    • Tampering: in the tamper lab, you changed a claim and verification failed even though you never touched the signature. Why?
    • Lifetime: what specifically gets worse with a 30-day token? What specifically gets worse with a 5-minute token?
  3. Write a token policy for Pelican Tours' guide app. The facts: nine guides, phones on boats, spotty signal; Saturday 2 PM a logged-in phone is stolen from a dock. Your policy must state: the access-token lifetime, whether/how it silently renews, and the exact kill-switch sequence — what the office does, in which system, and the longest time the thief realistically keeps access. Justify each number against the boat-and-bad-signal reality.
Hand in: the three answers + the token policy — in hw1/answers.docx.

Part 3 · Turn on real MFA

The concept — MFA: a second, different kind of proof. The three kinds: something you know (password), something you have (phone, key), something you are (fingerprint). Two of the same kind is not MFA.
  1. On GitHub: avatar → Settings → Password and authentication. The final setup for this course is GitHub Mobile as your preferred method, with SMS/Text message as a backup.
  2. If you see an Enable two-factor authentication button, GitHub requires you to begin with an authenticator app or SMS before GitHub Mobile becomes available. Choose SMS/Text message for this initial enrollment, enter your phone number, and confirm the code GitHub texts you.
  3. Install the GitHub Mobile app, sign in to the same GitHub account, and allow push notifications. Once 2FA is enabled and you are signed in to the app, GitHub Mobile appears under Two-factor methods.
  4. Back on Password and authentication, set Preferred 2FA method to GitHub Mobile. Keep SMS/Text message configured as the backup method. If the page already said Enabled, add whichever of these two methods is missing before setting GitHub Mobile as preferred.
  5. Under Recovery options, view your recovery codes and save them somewhere private — a password manager, or printed and put away.
  6. Sign out, sign back in, and confirm the second factor is actually asked for.
  7. Screenshot the settings page showing two-factor is enabled → save as screenshots/hw1-mfa-enabled.png.
  8. Answer two questions in hw1/answers.docx, one short paragraph each:
    • Classify: password, the login code texted to your phone, fingerprint — which of the three factor kinds is each?
    • Rank: order these from weakest to strongest against an attacker using today's AI tools — SMS code · TOTP app · voice verification · passkey/hardware key — and justify your top and bottom pick. Hint: which can an AI imitate, and which would it have to physically possess?
⚠️ Recovery codes are secrets. Never screenshot, paste, or commit them — even to a private repo. Your screenshot must show only the "enabled" state: no codes, no QR codes.
Hand in: screenshots/hw1-mfa-enabled.png + the two answers in hw1/answers.docx.

Part 4 · Plan for the guide who quits

The concept — identity lifecycle: every identity system lives or dies on three moments — joiner (grant exactly what the role needs), mover (adjust, don't accumulate), leaver (revoke everything, fast, completely). Forgotten leaver access is how breaches happen.

The scenario: Friday 5 PM, peak season, one of Pelican Tours' nine guides quits on bad terms, effective immediately. The ex-guide still has: a login to the admin panel, the shared photo-archive password, and membership in the group chat where customer manifests are posted.

  1. In hw1/answers.docx, build a table comparing two worlds:
    • World A — every tool has its own accounts (or shared passwords).

  2. One row per access the ex-guide holds — admin panel, photo archive, group chat at minimum; add anything else a real guide would have accumulated. For each row and each world: what must be done, who does it, in which system, and what happens if that row is forgotten.
