Homework 4 · One Master Copy

Your HW3 chatbot leaves your laptop. Deploy it to real AWS object storage, turn its documents into one shared, versioned master copy, break it on purpose, bring it back, and share a private draft the right way.

IndividualSubmit your GitHub repo linkRuns in your AWS Academy Learner Lab — no credit card, no new accounts

The scenario, continued. Your business from HW3 just hired a second employee. Two problems appeared overnight: the bot's policy documents are hard-coded into the page (so every policy change means editing code and redeploying), and "the website" currently means "whatever is running on your laptop." This week you fix both the way real teams do: the site moves to cloud object storage, and the documents become one master copy in the cloud that any authorized teammate can update — with version history as the safety net.

Before you start

  • You need your finished HW3 chatbot. If yours isn't working yet, the HW4 starter is a working baseline — but your HW3 features (threshold, system prompt, context limit) are still expected in what you deploy.
  • Start your AWS Academy Learner Lab exactly as in the project setup guide: Start Lab, wait for the green dot, then open the AWS Console. Confirm the region says N. Virginia (us-east-1) — everything in this homework happens there.
  • Create a new public GitHub repo cen4086-hw4 and clone it. Copy your HW3 index.html into it — you'll modify it here.
Learner Lab facts to plan around: the lab session ends after a few hours (click Start Lab again — your bucket and objects survive between sessions). Bucket names are globally unique across all of AWS, so use the pattern cen4086-hw4-yourinitials-mmdd. S3 for this assignment costs a few cents of your lab credits at most.

Part 1 — From laptop to cloud

  1. In the AWS Console, open S3 → Create bucket. Name: cen4086-hw4-yourinitials-mmdd. Region: US East (N. Virginia). Uncheck "Block all public access" and tick the acknowledgment — this bucket is going to be a public website on purpose.
  2. Open the bucket → Properties → scroll to Static website hosting → Edit → Enable → Index document: index.html → Save. Note the bucket website endpoint URL it shows you.
  3. Objects tab → Upload → your index.html.
  4. Public buckets still deny everything until a policy says otherwise. Go to Permissions → Bucket policy → Edit, and paste this — replacing YOUR-BUCKET-NAME in both places. Notice what it does not allow: it grants public read on exactly two objects, not on /*: 
    {
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "PublicReadExactlyTwoObjects",
    "Effect": "Allow",
    "Principal": "*",
    "Action": "s3:GetObject",
    "Resource": [
      "arn:aws:s3:::YOUR-BUCKET-NAME/index.html",
      "arn:aws:s3:::YOUR-BUCKET-NAME/docs.json"
    ]
  }]
}
  5. Open the website endpoint URL in a private/incognito window. Your chatbot is now served by object storage to anyone on the internet. Screenshot it with the URL visible.
What you just proved: a "website" can be nothing but objects behind HTTP — no server you manage, no OS to patch. Compare that to the Pelican Tours EC2 instance: which side of the responsibility line did all the patching duties just move to?

Part 2 — One master copy

  1. Right now your documents are trapped inside index.html. Create docs.json — just your DOCS array, as JSON (the starter's docs.json shows the exact shape).
  2. Change your index.html to fetch it instead of hard-coding it (the starter shows the change — about four lines, including a visible error message if the file is missing; you will want that message in Part 3): 
    // before:  const DOCS = [ …hard-coded array… ];
// after:   fetch the master copy that lives NEXT TO this page in the bucket
const DOCS = await (await fetch('./docs.json')).json();
  3. Upload both files to the bucket (overwriting index.html), reload the site, and confirm the bot still answers.
  4. Now the payoff. Without touching index.html: download docs.json, change one policy (shorten the return window, add an 11th document — your choice), and upload it back. Reload the live site and ask about that policy. The deployed bot changed and no code did. Screenshot the same question before and after.
This is Session 6's whole argument in one exercise. If each teammate kept their own copy of the policies, every edit would fork the truth — someone answers customers from the stale copy. One master copy in shared storage means everyone (including the bot) reads the same current version. What email attachments were to documents, hard-coded arrays are to application data.

Part 3 — Break it, then time-travel

  1. Bucket → Properties → Bucket Versioning → Enable. From this moment, S3 keeps every version of every object instead of overwriting.
  2. Upload one more revision of docs.json (another small policy change) so history has something in it.
  3. Now the disaster, on purpose: Objects → select docs.json → Delete (type the confirmation). Reload your live site: the bot should now show its missing-documents error. Screenshot the broken bot.
  4. Time-travel: in the Objects list, flip on Show versions. You'll see your docs.json versions still there, topped by a delete marker. Select the delete marker → Delete it. Reload the site — the bot is back, answering from the latest surviving version. Screenshot the version list and the healed bot.
  5. In your write-up, answer precisely: S3 keeps multiple replicas of every object across availability zones — why didn't that save you here, and what did instead? (You answered this on paper on the midterm; now you've run it.)

Part 4 — Share a private draft, properly

  1. Upload private-draft.txt — a one-paragraph "unannounced sale" policy draft. Try to open it in your incognito window via the website endpoint: Access Denied. Why? Reread your Part 1 bucket policy — the draft isn't one of the two allowed objects. That denial is least privilege working, not a bug. Screenshot it.
  2. Now share it without making it public and without creating an account for anyone: select the object → Actions (or Object actions) → Share with a presigned URL → set an expiry (e.g., 60 minutes) → create, and open the long URL in incognito: it works. Screenshot it.
  3. Write-up, three short answers: (a) Using Session 6's tool-selection lens, when is link-sharing the right model and when do you need account-based sharing with real identity? (b) What are the credentials hiding inside that long presigned URL, and what does its expiry remind you of from the token deep-dive? (c) Learner Lab twist: your lab session's credentials are themselves temporary — what happens to your presigned link after the lab session ends, even if you set a longer expiry, and why is that the same lesson?

Part 5 — The collaboration memo (no console)

Your two-person business is about to become four people and someone said the words "let's have a weekly all-hands." Write memo.md:

  1. Price the meeting with Session 6's formula: 4 attendees × a $40/hour loaded rate × 45 minutes, weekly — what does that meeting cost per year? Propose the async alternative for at least half of its agenda (shared doc + comments on your one master copy) and state what the remaining synchronous half is genuinely for.
  2. Pick the stack: using the six selection criteria from Session 6 (cost and free-tier edges, identity/SSO fit, data export, integration, and the rest), choose chat + docs + meetings tooling for the four-person team, one sentence of justification each.
  3. Price the bucket honestly: your site is roughly 1 MB and might transfer 5 GB out in a busy month. Using the course calculator's model prices (S3 $0.023/GB-month; transfer free for the first 100 GB), what does hosting cost per month? Compare it to three seats of a $6/user/month collaboration SaaS, and answer like a consultant: what is that SaaS seat price actually buying, given that the storage itself is nearly free?
