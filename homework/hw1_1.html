Trace a real SSO login

Goal: turn “Sign in with…” into a sequence you can explain

Choose a site you genuinely use that offers Sign in with Google, GitHub, Apple, or Microsoft. Your task is to observe one complete login and identify who controls each step.

Protect secrets while you observe. Record only domain names such as github.com or canva.com. Do not copy full URLs, query strings, authorization codes, cookies, tokens, passwords, or screenshots of the sign-in form into your repository. A URL can contain a short-lived secret even when it looks harmless.
If login seems to happen instantly: you probably already have a session with the identity provider. That is still a valid observation. Note that the provider reused an existing session. If you want to see the password/MFA step, use a private window only if you are comfortable signing in there; do not log out of an important account just for this assignment.

Look inside the token

In Part 1, “login result returned” was one step. Now zoom in. Complete the Inside the SSO Token interactive—journey, anatomy, tamper lab, lifetimes, and kill switch—then answer:

  1. The three roles. For your Part 1 login, who mints the identity token, who carries the front-channel response, and who verifies the token? Use one sentence each.
  2. The tamper question. You changed a claim and verification failed even though you did not edit the signature. Explain why in plain English.
  3. The lifetime trade-off. What specifically gets worse with a 30-day access token? What specifically gets worse with a 5-minute token?

Turn on real MFA

Your GitHub account guards your project repository, journal, and course deliverables. Protect it like it matters.

  1. In GitHub, open Settings → Password and authentication. Use GitHub Mobile or SMS/Text message — either is fine for this course.
  2. If an Enable two-factor authentication button appears, click it and choose SMS/Text message as the setup method (the wizard suggests an authenticator app first — choose SMS instead). Enter your phone number and confirm the code GitHub texts you.
  3. For GitHub Mobile (push approval on your phone instead of texted codes): install the GitHub app and sign in. It appears under Two-factor methods automatically once 2FA is on, and you can set it as your Preferred 2FA method.
  4. If the page already says Enabled — GitHub requires 2FA for many accounts — set your Preferred 2FA method to GitHub Mobile or SMS/Text message and confirm that method works.
  5. Store recovery codes (under Recovery options) privately in a password manager or printed in a secure place.
  6. Sign out, sign back in, and verify the second factor works before assuming setup is complete.
Recovery codes are credentials. Never screenshot, paste, or commit them—even to a private repository. Your submitted screenshot must show only that 2FA is enabled, with no codes, QR codes, email address, or other unnecessary personal data visible.

Explain what you protected

  1. Factor categories: classify a password, the login code texted to your phone, and a fingerprint as knowledge, possession, or inherence.
  2. The AI attacker: rank SMS code, TOTP authenticator app, voice verification, and passkey/hardware key from weakest to strongest against an AI-equipped attacker. Justify the strongest and weakest.
Hand in: screenshots/hw1-mfa-enabled.png with no secrets or unnecessary personal data, plus the two short explanations in hw1/answers.docx.

Plan for the guide who quits

Friday, 5 PM, peak season. One of Marissa’s nine guides quits on bad terms, effective immediately. The ex-guide’s phone still has a login to the future admin panel, the shared photo-archive password taped to the office wall, and membership in the “Guides” group chat where tomorrow’s customer manifests are posted.

Compare two identity worlds:

  • World A · accounts live in each app. Every tool has a separate guide account—or worse, a shared password.
  • World B · one central identity. Every tool trusts one identity provider and receives centrally managed users/groups.

Create a table in your Word document with one row per access type. For each world, state what must be done, who does it, which system they use, and the consequence if that row is forgotten.
