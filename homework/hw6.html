Homework 6 · Same Contract, Real Cloud

Move the support tools from local JSON files to a real AWS service—without changing what connected MCP clients see—then prove authorization failures, graceful degradation, persistence, logs, alarms, and measured-service cost.

IndividualSubmit your GitHub repo linkAWS Academy Learner Labus-east-1Fake data only

The scenario, continued. HW5's MCP server works—but only while your laptop is awake, and its tickets live in one local JSON file. Your business hires a remote support worker. Replacing the entire AI integration would be tight coupling. Instead, keep the same three MCP tool names and input schemas while moving the order/ticket implementation behind a cloud REST service.

AI host → local MCP adapter → HTTPS → API Gateway → Lambda → DynamoDB
           same contract                         ↘ CloudWatch logs + alarm

The adapter still publishes get_order_status, get_ticket, and create_support_ticket. The clients do not need to know whether the work happens in a JSON file, Lambda, a container, or tomorrow's vendor platform. That is loose coupling you can test, not just define.

What is—and is not—moving: the MCP server remains local because the lesson is the service boundary behind its tools. You are not trying to force a stateful MCP transport into Lambda. You are turning ordinary cloud services into reusable AI tools through a thin adapter.

One successful write, end to end

  1. You call create_support_ticket in the Inspector.
  2. The Inspector's MCP client sends a structured call over stdio to local server-cloud.mjs.
  3. The local adapter refuses immediately if confirmed is not true. If approved, it sends JSON over HTTPS with the demo token.
  4. API Gateway matches POST /tickets and invokes Lambda.
  5. Lambda independently rechecks the token, confirmation, order, and issue length.
  6. Lambda writes one DynamoDB item and emits a sanitized CloudWatch log.
  7. The JSON result travels back through API Gateway → adapter → MCP server → Inspector.

Part 1 — Persistent state: DynamoDB + Lambda

Goal: create the cloud data and compute layers. Do not create API Gateway routes until both layers are configured.

1A · Create the ticket table

  1. AWS Console → DynamoDB → Create table.
  2. Table name: cen4086-hw6-tickets-YOURINITIALS.
  3. Partition key: ticketId, type String. Keep default settings / on-demand capacity. Create the table and wait for Active.

1B · Create the function

  1. Lambda → Create function → Author from scratch. Name: cen4086-hw6-support-YOURINITIALS; runtime: the latest supported Python 3 runtime.
  2. Expand permissions → Use an existing role → select LabRole. Create the function.
  3. Replace the default editor contents with the starter's lambda_function.py. Click Deploy.
  4. Configuration → Environment variables → add TABLE_NAME with your exact DynamoDB table name.
  5. Generate a random temporary token locally—do not invent a short password: 
    node -e "console.log(require('node:crypto').randomBytes(24).toString('base64url'))"
    Add it as Lambda environment variable DEMO_TOKEN. Keep the value private; you will put the same value in a local ignored .env file.
Part 1 is done when: the DynamoDB table says Active; Lambda shows lambda_function.py as deployed; and Lambda Configuration → Environment variables lists TABLE_NAME and DEMO_TOKEN. Do not screenshot the token value.
Evidence: one screenshot of the active table configuration and one of the Lambda overview/configuration with secret values hidden.

The function checks the token, validates the route and arguments, rechecks confirmation and order existence, writes one ticket, and logs only IDs/timestamps. AWS Lambda automatically sends its invocation output to CloudWatch Logs; Part 4 makes that visible.

Part 2 — Publish the REST service and reconnect MCP

Goal: expose the Lambda through three precise HTTP routes, configure the local adapter, and prove one ticket survives outside the Lambda process.

2A · API Gateway

  1. API Gateway → Create API → HTTP API → Build. Name it cen4086-hw6-support-api-YOURINITIALS.
  2. Add your Lambda function as the integration.
  3. Create these three routes, all using that integration:
    • GET /orders/{orderId}
    • POST /tickets
    • GET /tickets/{ticketId}
  4. Use the $default stage with auto-deploy. CORS is not needed: the caller is a local Node process, not browser JavaScript.
  5. Copy the API's base Invoke URL—the https://…execute-api.us-east-1.amazonaws.com value with no route appended.

2B · Point the adapter at AWS

  1. Copy .env.example to a file named .env:
# macOS / Linux
cp .env.example .env

# Windows PowerShell
Copy-Item .env.example .env
  1. Open .env in a plain-text/code editor. Replace both placeholders; do not add the route /tickets to the base URL:
SUPPORT_API_URL=https://YOUR-ID.execute-api.us-east-1.amazonaws.com
SUPPORT_API_TOKEN=YOUR_RANDOM_TOKEN_FROM_PART_1
  1. Run git status. .env must not appear; the provided .gitignore excludes it. Run git check-ignore .env for positive proof.
  2. Run the direct boundary test, then the Inspector: 
    npm run test:boundary
npm run inspect
    The boundary test must report 401 UNAUTHORIZED for a wrong token and 409 CONFIRMATION_REQUIRED for an unconfirmed direct write. It never sends a successful write.
  3. Before the successful call, DynamoDB → your table → Explore table items should show zero items. In the Inspector, call get_order_status for ORD-1002. Then call create_support_ticket once for a known order with confirmed: true. Copy the returned ticket ID and read it back through get_ticket.
  4. DynamoDB → your table → Explore table items. Confirm exactly one ticket exists and matches the tool result.
Part 2 is done when: the direct test prints a 401 UNAUTHORIZED line, a 409 CONFIRMATION_REQUIRED line, and then its final PASS line; Inspector returns one real order, creates one approved ticket, and reads it back; and the identical ticket ID is the table's only item. If troubleshooting created extra fictional tickets, delete those test items and perform one clean final run before capturing evidence.
Evidence: boundary-test output, the three Inspector calls, and the matching DynamoDB item. Mask the API hostname if you prefer; the token must never be visible.

Part 3 — Prove both boundaries

Goal: demonstrate that bypassing the local MCP safety check still does not bypass the cloud service's safety check.

Your system has two enforcement points on purpose:

TestWhere it should stopEvidence
Inspector write with confirmed: falseLocal MCP adapter; it should make no HTTP request.Tool returns CONFIRMATION_REQUIRED; adapter unit test proves fetch was never called.
Direct HTTP write with the correct token but confirmed: falseLambda; a caller bypassed MCP, so cloud logic must still refuse.npm run test:boundary returns HTTP 409.
Direct request with a wrong tokenLambda authentication check before database access.Boundary test returns HTTP 401; table unchanged.
Approved write with correct tokenPasses both boundaries.HTTP 201 reaches Inspector; one DynamoDB item exists.

In your write-up, explain why the MCP prompt, tool description, local check, API token, Lambda validation, and DynamoDB permissions are not interchangeable controls. Then answer: if an attacker steals the demo token, which controls still limit the blast radius—and which production control should replace the shared token?

The important failure: “the model was told not to” is never authorization. A client can be buggy, compromised, or bypassed. The system that owns the side effect must validate the action under its own permissions.
Part 3 is done when: your table shows no item from either rejected direct request, exactly one item from the approved Inspector request, and your explanation identifies both the local and cloud enforcement points.
Evidence: one four-row table using the columns above, plus the adapter test, direct boundary-test, approved call, and DynamoDB item screenshots.

Part 4 — Operate it: logs, one alarm, one outage

Goal: observe one healthy event, turn one real error into an alarm, and prove the MCP tool reports a dependency outage honestly and then recovers.

4A · Read useful logs without leaking data

  1. Lambda → Monitor → View CloudWatch logs. Find the successful ticket_created event. It should contain request/ticket/order IDs and a timestamp—but not the issue text or demo token.
  2. Find the rejected-token warning. Explain why logging “unauthorized request” is useful but logging the supplied credential would create a new incident.

4B · Make an error alarm fire for real

  1. CloudWatch → Alarms → Create alarm → select your Lambda's Errors metric. Statistic: Sum; period: 1 minute; threshold: greater than or equal to 1. Name: cen4086-hw6-lambda-errors-YOURINITIALS. No notification action is required for this lab.
  2. Back in Lambda, create a console test event named intentional-alarm-test with exactly:
{"forceError": true}
  1. Run it once. The test must fail with Intentional HW6 alarm test. Wait for the metric period and refresh until the alarm enters ALARM. Screenshot the alarm and matching log stack trace.

4C · Graceful degradation

  1. Temporarily change SUPPORT_API_URL in .env to http://127.0.0.1:1, reconnect the Inspector, and call get_order_status.
  2. The tool must return an honest SERVICE_UNAVAILABLE error. It must not invent an order status or claim the action succeeded. Capture it, then restore the real URL and prove recovery with one successful read.
  3. Write a three-step fallback for a customer-facing AI host: what it tells the customer, what (if anything) it queues, and when a human is paged. Distinguish read failure from an uncertain write—blindly retrying a write can create duplicates.
Part 4 is done when: you have a sanitized success log, rejected-auth log, intentional error stack, alarm in ALARM state, SERVICE_UNAVAILABLE result, and a successful read after restoring the URL. CloudWatch may take 1–3 minutes to update.
Evidence: six clearly named screenshots matching the six items above, with no token or customer issue visible.

Part 5 — Loose coupling, least privilege, and cost memo

Goal: explain why the stable contract matters and evaluate the lab architecture as a production consultant would.

Write a one-page consultant memo with these four sections:

  1. Contract comparison. Put HW5 and HW6 Inspector screenshots side by side. Confirm the three tool names and input schemas stayed the same; list what changed behind them. Explain why an MCP-capable second client can discover the same service without another bespoke integration.
  2. Least privilege. LabRole is broad because it supports many labs. Design the production Lambda policy in words: which exact DynamoDB actions on which exact table ARN, and which log actions, would it need? State why API clients should never receive AWS credentials or direct table access.
  3. Production identity. Explain why the shared x-demo-token is acceptable only for fictional, short-lived lab traffic. Recommend an API Gateway authorizer/OAuth identity design that gives each caller revocable identity and scopes read vs. write.
  4. Measured-service math. Ignore free tiers and use these course model prices for one million ticket creations in a month: HTTP API calls = $1.00/million; Lambda requests = $0.20/million; Lambda compute = $0.0000166667 per GB-second; DynamoDB writes under 1 KB = $0.625/million. At 128 MB and 100 ms average duration, calculate total GB-seconds and total monthly cost before logs/storage/transfer. Compare it with one t3.micro running 730 hours at $0.0104/hour. Name what the always-on VM buys and what it wastes at quiet traffic.
Cost self-check: one million × 0.1 seconds × 0.125 GB = 12,500 GB-seconds. Your four serverless line items should total only a few dollars. MCP Inspector itself used no model; a production AI host would add separate token/inference charges.
Part 5 is done when: the memo contains four labeled sections, shows every cost calculation, distinguishes lab token from production identity, and names exact DynamoDB actions/resources for a least-privilege role.

Cleanup — part of the grade

Goal: remove the public endpoint, data store, compute, and alarm only after all required evidence is safely in your repo.
  1. Capture all evidence first. Record the Learner Lab budget again.
  2. Delete the API Gateway API, Lambda function, DynamoDB table, and CloudWatch alarm. Delete the homework log group if the lab permits it; otherwise note that it remains.
  3. Verify the named resources no longer appear, end the lab, and add a cleanup table to your write-up: resource, deleted at, final state, recoverable or not.
  4. Keep .env local until grading artifacts are finished, then delete it from your computer. Never commit it. If it ever reached GitHub, deleting the file in a later commit is not enough—rotate the token immediately and report the exposure.
Why delete instead of leaving a live grader URL? The grader needs your code, logs/screenshots, and test evidence—not a shared secret. Cleanup proves you can manage cloud resources through their full lifecycle and prevents an unnecessary public endpoint from lingering.
Cleanup is done when: searches in API Gateway, Lambda, DynamoDB, and CloudWatch show the named resources are gone (or the restricted log group is documented), the final budget is recorded, and the Learner Lab is ended.
