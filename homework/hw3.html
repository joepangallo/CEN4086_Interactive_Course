Homework 3 · The Chatbot That Cites Its Sources

Build a working, grounded FAQ chatbot that runs entirely in your browser — then prove it doesn't make things up, and price what it would cost to run for real.

IndividualSubmit your GitHub repo linkEverything free — no API keys, no accounts, no credit card

The scenario. You are building the product-support chatbot for a small business you invent (a bakery, a dive shop, a board-game café — your choice, but make it yours). The bot must answer customer questions only from the business's own policy documents, cite which document each answer came from, and — this is the hard part — refuse to answer when the documents don't contain the answer, instead of confidently making something up.

You are going to build the real thing. Not a diagram — a chatbot that runs in your browser using a genuine machine-learning embedding model, the same retrieval technique used by production AI systems.

Setup — from nothing to running (15 minutes)

Five steps. Each one tells you what you're doing, exactly what to type, and what you should see before moving on. If something doesn't match, jump to the troubleshooting table at the end of this section — your problem is almost certainly in it.

Step 1 — Open a terminal and check your tools · ~2 min

The terminal is a window where you type commands instead of clicking. This whole assignment needs it exactly twice — both times right here in setup.

  • macOS: press ⌘ + Space, type Terminal, press Enter.
  • Windows: press the Windows key, type PowerShell, press Enter.

Type each line below and press Enter after each. You're asking two tools to identify themselves:

git --version
python3 --version

You should see: a version number from each, like git version 2.44.0 and Python 3.12.4. Any version is fine — you only care that they answer.

  • Git missing? Install it from git-scm.com/downloads, accept every default, then close and reopen the terminal and try again.
  • Python missing? On Windows, first try py --version — Python often lives under that name. If it's truly absent, install from python.org/downloads and check the "Add python.exe to PATH" box on the installer's first screen — skipping that checkbox is the #1 cause of "python is not recognized." Then close and reopen the terminal.

Step 2 — Create your GitHub repository and clone it · ~4 min

The repository is where your work lives, and its link is the one thing you submit. You create it on the GitHub website first, then clone it — cloning downloads a linked copy into a folder on your machine, so the changes you make locally can be pushed back up.

  1. Sign in at github.com → click the + in the top-right corner → New repository.
  2. Repository name: cen4086-hw3 · visibility: Public · check "Add a README file" → click Create repository.
  3. On your new repo's page, click the green Code button and copy the HTTPS URL it shows.
  4. Back in the terminal, run these two lines, pasting your URL in place of the example: 
    git clone https://github.com/YOUR-USERNAME/cen4086-hw3.git
cd cen4086-hw3
    The first line downloads the repo into a new folder named cen4086-hw3. The second moves your terminal into that folder — everything else in setup happens from there.

You should see: Cloning into 'cen4086-hw3'... ending in done.

Step 3 — Put the starter app in your repo folder · ~2 min

The starter is a single self-contained HTML file. Get it into the folder you just cloned:

  1. Click this link: ⬇ Download the HW3 starter (zip). It lands in your Downloads folder.
  2. Unzip it — macOS: double-click; Windows: right-click → Extract All… — and move the index.html inside into your cen4086-hw3 folder.
Never use File → Save Page As… on an open tab. Browsers save a snapshot (".mhtml" / "Webpage, Complete") — a rewritten file full of MultipartBoundary and =3D garbage that will never run. If your bot ever renders as a wall of that text, you saved a snapshot: delete it and use the download link above.

You should see: ls (macOS) or dir (Windows) in the terminal lists index.html and README.md — and if you open index.html in a text editor, its very first line is <!DOCTYPE html>.

Step 4 — Start a tiny local web server · ~1 min

Why not just double-click the file? Because a page opened straight from disk runs as file://, and browsers put those pages in a security straitjacket — they're not allowed to fetch the AI model. Served over http://, they are. Python ships with a one-line web server for exactly this. From inside the cen4086-hw3 folder, run:

python3 -m http.server 8089

Windows: py -m http.server 8089 (or python — whichever name answered in Step 1).

You should see: Serving HTTP on :: port 8089 — and then the terminal appears to freeze. That's it working. Your machine is now a tiny web server: localhost is your machine's address as seen from itself, and 8089 is the door number the server is listening at. Leave this window open the whole time you work; when you're done for the day, stop the server with Ctrl+C.

Now open http://localhost:8089 in your browser.

Step 5 — Watch the model arrive, then talk to your bot · ~5 min

  1. On first load, the status line reports it's downloading the embedding model (about 25 MB) from a free public CDN. This is a one-time cost — the browser caches the model, so every later reload is instant. On slow Wi-Fi, give it a minute or two; on fast Wi-Fi it may flash to ready almost instantly. The model goes into the browser's cache, not your Downloads folder — don't expect to see a download bar.
  2. When the status line says ready, type "can I return a wetsuit?" and press Send.
  3. Definition of done: the bot answers from the sample surf-shop documents, names which document it used, and shows a similarity score. See all three and setup is finished — every part you build below extends this one file.
What just happened? Your browser downloaded and ran a real neural network (all-MiniLM-L6-v2, an embedding model) — inference, on your own hardware, for free. Nobody's server saw your question. Keep that in mind for Part 5, where you'll price what this would cost if it did run on someone's server.

Part 1 — Make it your business

The starter ships with six sample documents for a fictional surf shop. Replace them.

  1. Invent your business and write ten short policy/product documents (2–4 sentences each) in the DOCS array at the top of the file: returns, shipping, hours, warranties, a couple of product descriptions — whatever a real customer would ask about. Give each a clear title.
  2. Reload and ask five questions a customer would actually type — phrased differently from your documents' wording. Watch which document the bot retrieves and the similarity score it displays.
  3. In your write-up (see deliverables), explain in your own words why the bot can match "do you take stuff back" to a document titled "Return Policy" even though they share no words.
Where would these live in production? The document files belong in object storage (cheap, whole-file reads over HTTP, the citable source of truth). Their embeddings belong in a vector database built for nearest-in-meaning search. Your DOCS array plays both roles at once — say so in your write-up, and name which real service type would take over each role.

Part 2 — The system prompt (persona & boundaries)

The starter has a System prompt box that currently does nothing. Wire it up:

  1. Make the bot's replies open with the persona defined in the box (e.g., greeting style, sign-off, tone words) — the reply template should read its values instead of hard-coding them.
  2. Add at least one boundary rule the bot enforces from configuration: for example, a list of off-limits topics that always get the polite refusal, no matter what the retrieval finds.
  3. Demonstrate it: change the system prompt text only (no other code) and screenshot the same question answered in two different personas.
Why this matters: this is exactly how a vendor ships one shared model to hundreds of businesses that each get their own bot personality — the configuration layer, not a different model. You built the mechanism you studied in Session 2.

Part 3 — The context window (make it forget, on purpose)

  1. The starter keeps the whole chat history. Add a CONTEXT_LIMIT (a character budget). Before answering, trim the oldest turns until the kept history fits the budget — exactly how a real model's finite attention window behaves.
  2. Show the user what's happening: display how many characters of history are "in the window" and gray out the messages that fell outside it.
  3. Demonstrate the failure you studied: tell the bot your name, paste two long paragraphs of filler, then ask "what's my name?" Screenshot the bot failing — and explain in your write-up why it still answers document questions perfectly even after "forgetting" your name. (Hint: which store did your name live in, and which store do the documents live in?)

Part 4 — The hallucination test (the whole point)

  1. The starter answers with the closest document no matter how bad the match is. Fix that: add a similarity threshold — below it, the bot must reply "That's not in my documents — let me connect you with a person." and cite nothing.
  2. Calibrate it: try thresholds until your five real questions from Part 1 still answer, but nonsense does not.
  3. Run this adversarial test protocol and record all results in a table (question, top score, bot behavior, verdict):
    • three fair questions your documents do answer,
    • three questions your documents do not answer but sound plausible ("do you price-match?" when you have no price-match policy),
    • one question about a different business entirely.
  4. In your write-up: for a customer-facing bot, which is the worse failure — refusing a question it could have answered, or confidently citing the wrong policy? Connect your answer to the stakes-based verification idea from Session 6.
The bright line, restated: a bot that answers from retrieved documents with citations can be checked. A bot that answers from vibes cannot. Your threshold is the difference.

Part 5 — The cost memo (no code)

Your browser bot is free because your laptop does the inference. Real deployments pay one of two ways. Write a one-page memo, memo.md, pricing both for your business:

  1. Per-token API: assume $0.60 per million input tokens and $2.40 per million output tokens, and that an average exchange uses 900 input + 250 output tokens. Price a quiet month (600 customer questions) and a busy month (45,000 questions). Show the calculations.
  2. Always-on rented GPU: a g4dn.xlarge at $0.526/hour running 730 hours. Price the month — and note that the price doesn't change whether it answers 600 questions or 45,000.
  3. The recommendation: which usage pattern favors the API and which favors the rented GPU? At roughly how many exchanges per month do the two curves cross? Which cloud pricing trait from Week 1 does the API model exemplify, and which Week 1 warning does the always-on GPU deserve?
  4. Classify first: two short calls — (a) chat transcripts sometimes contain customers' personal details; if some customers are Canadian, what does that force about where transcripts are stored? (b) A teammate proposes letting the bot look up card numbers "to check payment status." Using the classification rule, say no properly: name the constraint and the design decision it forces.

Optional stretch — a real generator

Not required. If you want the bot to write answers instead of quoting documents: WebLLM (webllm.mlc.ai) runs a small open language model entirely in your browser — free, no key. Feed it your retrieved document plus the question and let it phrase the reply, keeping your citation and threshold logic. Note the model download is large (over 1 GB) and needs a recent GPU-capable machine. Your grounding, threshold, and citations must still work — the generator is a mouth, not a memory.
