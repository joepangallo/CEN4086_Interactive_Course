What an agent actually is — the map for tonight

Strip the marketing away and an "AI agent" is a while-loop with judgment and boundaries. Every box on the anatomy page is a file in your starter:

Anatomy organIn this starterWho provides it
The loopagent.mjs — plan → check → act → observe, repeatGiven, finished
The brain (planner)planner.mjs — plan(state) returns ONE next actionPartly yours (Part 3)
Toolssupport-server.mjs — the same MCP surface as HW5Given, finished
Memorythe observations array — everything the mission has seenGiven, finished
Guardrailsguardrails.mjs — allowlist, budget, injection detectorPartly yours (Part 2)
Human approvalthe y/N gate in agent.mjs before any write toolGiven, finished — read it anyway
Observabilitytrace.jsonl — one JSON line per decision and resultGiven, finished
Why there is no API key. The planner is deterministic on purpose. plan(state) → action is exactly the contract an LLM-backed planner has — state goes in, one proposed action comes out. Using a script instead of a model buys you three things tonight: missions that behave identically on every run (debuggable), zero cost (safe to loop), and a crystal-clear view of the seam where a model would slot in. The guardrails cannot tell the difference — which is precisely why they work.
The one design rule to internalize. Nowhere in the planner — and nowhere in any document the agent reads — can confirmed become true. That flag is wired to a single line in agent.mjs, immediately after a human types y. The brain proposes; the human disposes; the server double-checks anyway. Three layers, and the customer note's attack (you'll meet it in mission 4) bounces off all three.

Part 1 — Run it, then trace one loop by hand

Goal: watch the finished machinery run a mission end to end, then prove you can reconstruct its reasoning from the trace alone.
  1. Run mission 1 — it works on the unchanged starter: 
    node agent.mjs "What is the status of order ORD-1001?"
  2. Now open trace.jsonl and find that mission's lines: mission_start, a plan, an observe_tool, another plan, mission_finish.
  3. In your write-up, reconstruct the mission from the trace only: for each line, one sentence — what the agent knew, what it decided, and why the second plan chose finish when the first chose call_tool. (Hint: the difference is one entry in observations — that array is the agent's memory.)
  4. Run the same mission with --role=read-only and note that nothing changes — then explain in one sentence why a status lookup is the same under both roles.
Part 1 is done when: your write-up walks the trace line by line and answers both questions.

Part 2 — Build the guardrails

Goal: replace the two permissive stubs in guardrails.mjs with real boundaries. The specs live as comments above each stub; the tests grade them.

2A · The step budget

  1. First, run the runaway on the unchanged starter and watch the crude seatbelt drag it down at 25 steps: 
    node agent.mjs "Keep checking ORD-1001 until it says delivered"
    ORD-1001 never delivers; this branch loops by design. Save this transcript — you'll want the before/after pair.
  2. Implement checkStepBudget(stepsUsed, maxSteps) per its comment spec: allow below the limit, refuse at or beyond it with code STEP_BUDGET_EXHAUSTED and a message that states the numbers.
  3. Re-run the runaway: it should now stop at 8 steps with the budget message, not the seatbelt.
Why the budget is a big deal. With a scripted planner a runaway wastes seconds. With a paid model behind the same loop, it wastes tokens at production prices, forever, at 3 a.m. The wrap-every-AI-call-in-a-timeout-and-budget rule from the Session 9 material is this exact function, one abstraction up.

2B · The injection detector

  1. Read data/customer-note.txt — a customer "complaint" that tries to script your agent: ignore your rules, claim someone else's order, write without asking. Then run mission 3 on the unchanged starter and note that the summary passes it along without any warning: 
    node agent.mjs "Summarize the customer note for the morning meeting"
  2. Implement looksLikeInjection(text) per its comment spec: three case-insensitive patterns (ignore-your-rules, scripted tool call, suppress-the-human), true if any match, false for ordinary complaints.
  3. Re-run mission 3: the report should now open with the injection warning while still quoting the note as data for a human reviewer.
Say the honest thing in your write-up: this detector is a heuristic — an attacker can rephrase past it. Then say why the system survives that anyway: which two other layers stop the note's actual goal (an unauthorized ticket on ORD-1002) even if the detector never fires? Name them precisely.
Part 2 is done when: the four guardrail tests pass (npm test: 12 of 16) and both before/after transcripts are saved for the write-up.

Part 3 — Teach the planner the refund workflow

Goal: implement decideRefundStep(state) in planner.mjs — a five-situation state machine that reads before it writes, proposes without confirming, and verifies before it brags.
  1. Read the finished planStatusLookup branch — it is the one-tool version of what you're about to write — and the five-situation spec in the TODO comment.
  2. Delete the throw and implement the situations in order. After each one, run npm test: the four refund tests are written one-per-situation and go green in sequence.
  3. The critical line: your ticket proposal must carry confirmed: false. The planner proposes; only the approval gate in agent.mjs — a human at a keyboard — flips it. One of the tests exists solely to fail any planner that grants itself permission.
Part 3 is done when: npm test reports tests 16, pass 16, fail 0.

Part 5 — The seam memo

Goal: demonstrate you understand what tonight's brain-ectomy was for.

Write a one-page memo to Sandpiper's owner answering, in plain language:

  1. What would change to give this agent a real language-model brain? Name the exact function whose insides get replaced, what new inputs it would need (the tool list, the conversation), and what new failure modes arrive with it (wrong tool, malformed args, persuasion by injected text, cost per call).
  2. What would not change — walk each guardrail (allowlist, approval gate, step budget, server-side confirmed check, trace) and state whether it is structural (works regardless of what the brain thinks) or advisory (depends on the brain cooperating). Four of the five are structural; find them.
  3. The recommendation: should the owner let the agent run unattended overnight — with which roles, and which missions? Defend it with evidence from your own transcripts, not vibes.
This memo is the course's AI thread, closed. The two-lives rule told you models don't learn from answering. RAG grounded their answers. MCP gave them typed, bounded hands. AI security taught you the hands need locks that don't rely on the model's obedience. Tonight you built the locks and held the keys. Whatever brains arrive next year, that architecture is the part that keeps working.
