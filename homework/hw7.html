Homework 7 · Cutover Weekend

Disposition a small application portfolio with the 6 Rs, rehost the public front-desk site to Amazon S3 static website hosting, rehearse a real cutover — bad deploy, detection, rollback — with your own synthetic monitor, put operations guardrails on the bucket, estimate the monthly bill, and leave the account exactly as you found it.

The scenario, continued. Sandpiper Surf Supply's support tools now live in the cloud (HW6) — but the business's public front desk page still runs on the aging computer in the back office. It has hours, policies, and a status banner, and it goes dark every time that machine reboots. The owner has approved a migration weekend. You are the engineer who plans it, executes it, proves it with a monitor, and — because Session 11 taught you the bill is a metric too — prices it.

BEFORE   customer → office computer (serve.mjs on your laptop) → static files
AFTER    customer → S3 static website endpoint → the same static files
                    ↘ versioning (rollback net) · tags · lifecycle · monitor

No servers survive this migration. That is the point: a static site on S3 has no machine to patch, no process to restart, and no midnight pager for a dead power supply. Your job shifts from keeping a computer alive to operating a service — verifying deploys, planning rollback, tagging for cost, and reading the bill.

Read this first — what runs where

Three programs appear in this assignment, and only one of them moves to the cloud:

ProgramRole in the storyWhere it runsWhat happens to it
site/ (three files)The application being migratedLaptop → S3Rehosted. The files do not change; only where they are served from changes.
serve.mjsThe legacy on-prem serverLaptop onlyRetired at the end — the happiest of the 6 Rs.
monitor.mjsYour synthetic monitorLaptop onlyRetained. It verifies the old site, the new site, and every deploy after cutover.
Why the monitor is the star. Session 11's first law was you can't manage what you can't see. The monitor is the smallest possible "seeing" machine: it fetches the site like a customer, checks content — not just status codes — and returns an exit code a scheduler could act on. Notice that it takes any base URL. A monitor that only worked against one hosting setup would die in the migration; yours survives it, which is exactly what real operations teams demand of their tooling on cutover day.
Static hosting is not "less cloud." S3 static website hosting is a fully managed service: eleven-nines durability, no capacity planning, pay-per-use pricing. It cannot run server-side code — no Lambda-style logic, no database calls from the server — which is why the disposition step matters: rehost is only right for apps that truly are static. Choosing it blindly for everything is how migrations fail.

Part 0 — Preflight: feel the legacy

Goal: prove the starter works, meet the monitor, and experience the outage the migration exists to eliminate.
  1. In the starter folder run npm test. Every test should pass — if not, fix your Node version before continuing.
  2. Run npm start and open http://localhost:8080. The banner should read "All systems normal…" with release 1.0.0.
  3. In a second terminal run node monitor.mjs http://localhost:8080 — five PASS lines, SITE HEALTHY.
  4. Now stop the server (Ctrl+C) and run the monitor again. Watch it fail. In your write-up, answer: which checks failed, what did the exit code become, and who in a real business notices this outage first — the engineer or the customer?
  5. Restart the server for now; you retire it in Part 6.
Preflight is done when: tests pass, the monitor shows healthy against localhost, and your write-up describes the outage run.

Part 1 — Disposition: the 6-Rs table

Goal: decide what should happen to each item in Sandpiper's little portfolio — before touching the console, like a real migration team.

Sandpiper's owner lists six "systems." For each, choose exactly one of the 6 Rs and defend it in one or two sentences. Wrong-but-defended beats right-but-unexplained.

#SystemFacts you have
1Front-desk website (this assignment's site/)Three static files; no server-side logic; currently on the office computer
2Support ticket serviceYou already moved it: HW6 turned it into API Gateway + Lambda + DynamoDB
3The office computer itselfRuns the website and nothing else once tickets moved
4PayrollA spreadsheet the owner hates; several SaaS payroll products exist
5The fax-order workflowReceived two orders last year, both from the same customer, who also emails
6The point-of-sale terminal in the storeCertified hardware+software bundle from a vendor; contract renews in three years; works fine
The two traps. Every real portfolio contains something that should be retired (migrating it would be waste) and something that should be retained (moving it now is needless risk). Migration maturity is measured by what you don't move. If your table has no retire and no retain, look again.
Deliverable: the six-row disposition table with justifications, in your write-up.

Part 2 — Rehost: the site moves to S3

Goal: serve the identical files from S3 static website hosting and prove it with the same monitor that watched the legacy server.

2A · Create the bucket

  1. Console → S3 → Create bucket. Region us-east-1. Name it something globally unique, e.g. sandpiper-frontdesk-<initials><4 digits>.
  2. Under Block Public Access, uncheck "Block all public access" and acknowledge the warning. You are hosting a public website; this is the one situation where that checkbox is supposed to come off. Leave every other default.

2B · Turn on static website hosting

  1. Open the bucket → Properties → scroll to Static website hosting → Edit → Enable.
  2. Index document: index.html. Save, then copy the bucket website endpoint URL — it looks like http://BUCKET.s3-website-us-east-1.amazonaws.com. This is your site's new address.

2C · Upload and publish

  1. Objects → Upload → add the three files from site/ (index.html, style.css, status.json). Upload them at the top level of the bucket — not inside a site/ folder — so the paths match what the page expects.
  2. Attach the public-read policy: Permissions → Bucket policy → Edit → paste, replacing BUCKET-NAME with your bucket's exact name:
{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "PublicReadGetObject",
    "Effect": "Allow",
    "Principal": "*",
    "Action": "s3:GetObject",
    "Resource": "arn:aws:s3:::BUCKET-NAME/*"
  }]
}
Read the policy before pasting it — it is four decisions. Principal: "*" means anyone (a website's audience). Action: s3:GetObject means read only — the internet can download your files but can never upload, change, or delete one. Resource: .../* scopes it to objects in this one bucket. And there is no second statement: nothing here grants list, write, or delete to anyone. Least privilege isn't only for private systems; a public site is "everyone may GET, and that is all."

2D · Verify like an operator

  1. Open the website endpoint in your browser — same page, same banner, no laptop involved.
  2. Now the professional version: node monitor.mjs http://YOUR-ENDPOINT. All five checks must pass.
  3. Stop your local serve.mjs and run the monitor against the S3 endpoint again. It still passes — the site no longer needs your laptop. That single observation is the rehost, proven.
Part 2 is done when: the monitor reports SITE HEALTHY against the S3 website endpoint with your laptop server stopped.
Evidence: the monitor's full output against the S3 endpoint (paste the terminal text into your write-up), plus one screenshot of the bucket's Static website hosting panel showing the endpoint.

Part 3 — Cutover drill: ship it, break it, roll it back

Goal: rehearse the moment Session 12 called "the weekend everything switches": deploy a new release, verify it, then survive a bad deploy using versioning as the rollback net.

3A · Arm the rollback net first

  1. Bucket → Properties → Bucket Versioning → Edit → Enable.
  2. In your write-up, note the order of operations here: versioning goes on before the risky deploy, for the same reason seatbelts fasten before the car moves.

3B · Ship release 1.1.0 (the good deploy)

  1. Locally, edit site/status.json: change version to 1.1.0 and the banner to a message announcing the new cloud home (keep it fictional and professional). Commit.
  2. Upload the edited status.json to the bucket, overwriting the old one.
  3. Verify with a release expectation: node monitor.mjs http://YOUR-ENDPOINT --expect-version=1.1.0 — six checks, all passing. Note why the sixth check exists: "something responds" and "the new release is live" are different claims.

3C · The bad deploy (on purpose)

  1. Upload drills/broken-status.json to the bucket renamed as status.json (easiest: locally copy it over site/status.json, upload, then restore your local copy with git checkout -- site/status.json). This simulates the classic truncated-upload failure.
  2. Run the monitor with --expect-version=1.1.0. Read the output carefully: which checks still pass, which fail, and what does the failure message tell an on-call engineer at 2 a.m.?
  3. Open the site in your browser too — the page's own error handling shows a fallback banner. In your write-up: why is "fail visible" better than a blank page, and which of the two (monitor or page fallback) actually pages a human?

3D · Roll back with versioning

  1. Bucket → Objects → toggle Show versions. status.json now lists multiple versions, newest first.
  2. Select only the newest (broken) version of status.json and delete it. Deleting the newest version un-hides the previous good one — that is the whole rollback.
  3. Verify: node monitor.mjs http://YOUR-ENDPOINT --expect-version=1.1.0 — healthy again. Time yourself from "monitor red" to "monitor green" and record it.
Precision matters here. Deleting the object instead of the one broken version deletes the site's status file entirely (it leaves a delete marker). If that happens, don't panic: show versions, delete the delete marker, breathe, and write down what you learned about why rollback procedures are rehearsed.

3E · Write the cutover runbook

Turn what you just did into a numbered runbook another student could execute: pre-checks, deploy step, verification command, the exact monitor output that triggers rollback, the rollback steps, and the final verification. One page, imperative voice ("Run…", "If X, then Y"), in your repo as runbook.txt or runbook.pdf.

Part 3 is done when: your terminal history shows healthy 1.1.0 → broken → healthy 1.1.0 again, and the runbook exists.
Evidence: the three monitor outputs (good deploy, bad deploy, post-rollback) pasted in order into your write-up, your red-to-green time, and the runbook file.

Part 4 — Operate it: tags and a lifecycle rule

Goal: leave the bucket in a state an operations team could live with — attributable on the bill and unable to hoard old versions forever.

4A · Tags: who pays for this?

  1. Bucket → Properties → Tags → add two: app = sandpiper-front-desk and env = production.
  2. In your write-up, connect this to Session 11's chargeback lesson: with fifty untagged buckets, what question can the finance team not answer?

4B · Lifecycle: versioning without a hoarding problem

Versioning saved you in Part 3, but every old version is billed storage, forever, unless a rule says otherwise.

  1. Bucket → Management → Create lifecycle rule, e.g. expire-old-versions, applied to all objects.
  2. Add the action Permanently delete noncurrent versions after 30 days.
  3. Add the action Delete expired object delete markers or incomplete multipart uploads → incomplete multipart uploads after 7 days.
Why 30 days is a decision, not a default. A shorter window shrinks the bill; a longer one lengthens how far back you can roll. Rollback nets are insurance, and lifecycle rules are the premium you choose. In your write-up, state one scenario where 30 days would be too short.
Part 4 is done when: both tags exist and the lifecycle rule is Enabled with both actions.
Evidence: one screenshot of the tags and one of the enabled lifecycle rule's summary.

Goal: price the migrated site like the metric it is.

Assume Sandpiper's front desk sees this traffic every month:

  1. Work the arithmetic: total GET requests/month, total GB transferred out/month, total GB stored. Show every step.
  2. Using the course AWS Cost Calculator page and current public S3 pricing for us-east-1, estimate the monthly cost line by line: storage, requests, data transfer. State which line dominates and why the other two round to almost nothing.
  3. Compare honestly against the office computer it replaced — electricity, the occasional dead disk, and the outage every reboot caused. Cheap is good; predictable and visible is the deeper win. Say which mattered more here and why.
  4. Finish with one sentence on what would change if a viral post multiplied traffic by 100 overnight — for the S3 site, and for the office computer.
Deliverable: the cost memo (arithmetic + line estimates + comparison + viral-day sentence) as cost-memo.pdf, .docx, or .txt in your repo.

Part 6 — Cleanup: leave no trace

Goal: remove everything you created. A versioned bucket makes this a two-step operation — learning that is the exercise.
  1. Bucket → Empty → type the confirmation. Emptying a versioned bucket deletes all versions and delete markers, not just current objects — read the dialog and say so in your write-up.
  2. Then Bucket → Delete → type the bucket name.
  3. Confirm the S3 console shows no buckets of yours, record your lab budget's ending value next to the starting value from Part 0, and end the lab session.
  4. Retire the legacy: you never need serve.mjs again. Note the retirement in your write-up — system #3 from your disposition table just exited the portfolio for real.
Part 6 is done when: the bucket is gone, both budget readings are in the write-up, and the lab session is ended.
