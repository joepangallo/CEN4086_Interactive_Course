When Maya gets to the EC2 User data box, this is the full script she pastes.

#!/bin/bash
set -eux

APP_ZIP_URL="https://github.com/YOUR-GITHUB-USERNAME/pelican-tours-m1/archive/refs/heads/main.zip"

dnf install -y nodejs unzip

mkdir -p /opt/pelican
cd /opt/pelican
curl -fsSL "$APP_ZIP_URL" -o app.zip
unzip -o app.zip

APP_DIR=$(dirname "$(find /opt/pelican -name server.js | head -1)")

cat >/etc/systemd/system/pelican.service <<EOF
[Unit]
Description=Pelican Tours booking app
After=network.target

[Service]
WorkingDirectory=${APP_DIR}
ExecStart=/usr/bin/node server.js
Environment=PORT=80
Restart=always

[Install]
WantedBy=multi-user.target
EOF

systemctl daemon-reload
systemctl enable --now pelican

I used t3.micro because this is a small Node booking app with fictional class traffic. The app mostly serves simple HTML, JSON tour data, and small booking requests.

Maya's lost-state example

Maya books a tour with the fake name Maya Test. The booking appears. Then traffic moves to another instance after scaling/replacement. The booking is missing.

## Lost-state experiment
I booked a fake tour as Maya Test. The booking existed for the instance that handled it. After scaling/replacing instances, the booking was gone.

Reason:
The starter app stores bookings in memory. Memory belongs to one running EC2 instance. A second instance does not share that memory, and a replaced instance loses it completely.

Real fix:
Bookings need shared persistent storage. DynamoDB is the AWS service we will use later for that.

Jordan's broken-state example

Jordan's app lets a user add a fake pantry item, but the item is stored in memory. When the ALB sends traffic to another instance, the new item is missing.

## Broken-state experiment
I added a fake item named "Rice Bags - Test Item" through the Campus Pantry frontend.

What broke:
The item appeared when one instance answered, but disappeared when another instance answered.

Why:
The API stores added items in memory. Each EC2 instance has its own memory, so the two instances do not share the same item list.

Real fix:
The item list needs a shared database. DynamoDB or another managed database would make every instance read and write the same durable data.

## Lost-state experiment
I booked a fake tour as Maya Test. The booking existed at first. After scaling/replacing instances, the booking disappeared.

The booking disappeared because the starter app stores bookings in memory. Memory belongs to one EC2 instance. Another EC2 instance cannot see it, and a replacement instance loses it completely. A real fix is shared persistent storage such as DynamoDB.

## Build vs buy
I compared this custom build with a booking SaaS such as FareHarbor or Peek.

SaaS advantage:
The company gets booking features, hosting, updates, and support without managing EC2, load balancers, or deployments.

Custom build advantage:
The company gets more control over the exact workflow and data model.

Lock-in:
A SaaS tool may make it hard to leave later if all bookings and customer workflows live there.

Multi-tenancy:
A SaaS booking system usually serves many companies from one shared platform. My AWS build is just for this one client.

Recommendation:
For a tiny tour company that needs bookings immediately, I would start with SaaS. I would build custom only if the business has a special workflow or needs control the SaaS cannot provide.

PaaS lens:
The ALB and ASG managed health checks, traffic distribution, replacement, and capacity rules. A full PaaS would also hide more of the OS, runtime, deployment, and process management.
