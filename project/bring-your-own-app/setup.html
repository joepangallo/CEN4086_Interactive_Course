✅ Does your app qualify — and how to start

This track is for students who already have a mobile app or a website with an app component. Eligibility, the proposal (emailed right after Class 3), safety rules for working with a real app, and how each kind of app adapts to the milestones.

Eligibility

Your app qualifies if all of these are true:

  • It's yours. You wrote it (side project, club app, a previous course project), or it was a team project and you're working from your own fork and can modify every layer yourself. All course work is individual.
  • You have the source in a repository you control, and it runs today — on your machine, an emulator, or a device.
  • It has, or can gain, a server side. The milestones operate on backend components. A pure static site or a fully offline mobile app qualifies only if you're willing to build its minimal API backend in M1 (the sibling tracks' starter apps show how small that can be).
  • It can run on course infrastructure: your own machine with Docker, or the AWS Learner Lab. Apps hard-wired to a platform you can't replicate (e.g., entirely Firebase-dependent with no replaceable layer) need a plan for which slice you'll re-implement — put it in the proposal.
  • A course copy is possible. You can run a separate instance with test data, without touching any live version real people use.

Not sure it qualifies? Ask at Class 3 — borderline cases usually work with a small scope adjustment. If it isn't approved, you take the Pocket Cloud or Pelican Tours track instead, no penalty.

Safety rules for a real app

🔒 Non-negotiable:
  • Course copy only. All milestone work happens on a separate branch/fork and a separate deployment. Never point course experiments at a live version people use.
  • No real user data, ever. Test/generated data only in the course deployment, its database, its backups, and every screenshot. If a database dump contains real emails, names, or messages — it does not enter the course copy.
  • Secrets: production API keys, signing keys, and store credentials never go in the course repo. M3's secrets audit covers the course copy; rotate anything you find already leaked.
  • Game day (M3) is for the course deployment only. You destroy and restore the course copy — never a live system.
  • Platform rules still apply: LOCAL stays on localhost/LAN — no port-forwarding to the internet · AWS follows every Learner Lab cost/safety rule in its guide.
