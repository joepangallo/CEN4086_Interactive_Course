📌 TL;DR — what you're actually handing in Monday

Regardless of track, Class 4 wants five things sitting in one place (your repo, JOURNAL.md, and a screenshots/ folder):

  1. A working cloud deployment reachable by URL right now — not a screenshot from three days ago.
  2. Proof it survives a load balancer + Auto Scaling group — traffic rotating across instances, and one instance dying and getting replaced automatically.
  3. A journal with your sizing reasoning, your "what broke when I scaled" finding, and your budget numbers — in your own words, not just checkboxes.
  4. A build-vs-buy appraisal using real SaaS/PaaS/IaaS vocabulary.
  5. Everything scaled back down to desired capacity 1 before you close the laptop — an unattended budget burn is its own kind of failing grade.

The two sections below (🌊 starter app vs 📱 your own app) turn each of those five into exact, track-specific steps and an exact submission checklist. Pick your card.

🌊 Track A — Starting from Zero (Pelican Tours Starter App)

Marissa's booking app moves from "vanished web kid's laptop" to real AWS infrastructure, then proves it can survive a viral video. Full brief: Pelican Tours project page · Starter code: starter-app guide.

📦 Exactly what you submit Monday

  • Repo link containing the starter app (your fork/copy), your edited user-data.sh, JOURNAL.md, and a screenshots/ folder with every screenshot used as evidence.
  • A live URL — your Application Load Balancer's DNS name — that resolves to the working booking app at submission time.
  • Journal entry: budget & NIST trait — your first budget-page screenshot + one sentence naming who actually owns the hardware under your $100, and which NIST trait that illustrates.
  • Screenshot: working site + /api/health from your first single instance, plus one sentence on a decision you had to make (AMI, security group, instance profile) that a SaaS user would never see.
  • Sizing memo (written) — t3.micro's vCPU/RAM/burst-credit numbers and on-demand price/hr, plus the instance type (and its price) you'd move to at 10× traffic.
  • Screenshot: served_by rotating — several reloads of /api/health through the ALB DNS name showing different instance IDs answering.
  • Screenshot pair: self-healing — before/after of manually terminating one instance and the Auto Scaling group replacing it, with a timeline note (how long did it take?).
  • Lost-state journal entry — place a booking, scale/replace instances, show the booking is gone; explain why in-memory state breaks under elasticity and which AWS service(s) would fix it (you implement the fix in M4).
  • Phone screenshot — one tour booked from your phone over cellular data (not Wi-Fi) via the ALB URL, plus one sentence.
  • Build-vs-buy write-up (half page) — booking SaaS (FareHarbor, Peek, etc.) vs. your build, compared on cost model / control / lock-in / multi-tenancy, plus a PaaS-lens paragraph on what the ALB+ASG just did for you.
  • Cost ledger table — every resource currently running, its hourly price, and your measured burn for the week, plus a predicted M2 burn number.
  • Proof the budget is respected — Auto Scaling desired capacity back down to 1 before you submit.

🔧 Step-by-step build guide

# M1 target architecture
#
#   phone/browser ──▶ ALB (port 80) ──▶ Auto Scaling group (1–3 × t3.micro)
#                                        each runs starter-app via user-data
#
# Order of operations (console, us-east-1):
# 1. EC2 → Launch templates → create from your working single instance
#    (AMI: Amazon Linux 2023 · t3.micro · LabInstanceProfile · user-data)
# 2. EC2 → Target groups → HTTP:80, health check path /api/health
# 3. EC2 → Load balancers → ALB, internet-facing, 2+ AZs, your target group
# 4. EC2 → Auto Scaling groups → min 1 / desired 2 / max 3, attach target group
# 5. Visit http://ALB-DNS-NAME/api/health repeatedly → served_by rotates
# 6. Terminate an instance → watch ASG replace it (screenshot before/after)
# 7. Desired capacity → 1, then End Lab. Budget line in journal.
# Where user-data downloads your app from (pick ONE, before you launch):
#
# Option A — public GitHub repo (simplest, most reliable):
#   1. Push the starter-app folder to a public repo you create
#   2. Zip URL: https://github.com/YOUR-USERNAME/YOUR-REPO/archive/refs/heads/main.zip
#   3. Paste that into APP_ZIP_URL in user-data.sh
#
# Option B — S3 bucket in your lab account:
#   1. Zip the folder:  zip -r starter-app.zip starter-app
#   2. S3 → Create bucket → Upload starter-app.zip
#   3. Select the object → Presigned URL (12 hours) → paste into APP_ZIP_URL
#      (an instance relaunching after the URL expires will fail — Option A avoids that)

⚠️ Where students lose easy points

MistakeFix
Bookmarked an instance's public IPIPs change every time the lab session restarts instances. Use the ALB DNS name everywhere — journal, screenshots, submission link.
"user-data didn't run"It only runs on first boot. Check /var/log/cloud-init-output.log, fix APP_ZIP_URL, and relaunch a fresh instance rather than debugging a running one.
SSH open to 0.0.0.0/0Restrict port 22 to your IP only before you screenshot the security group — this is graded, not cosmetic.
Presigned S3 URL expiredIt's only valid 12 hours; any instance that relaunches after that fails silently. Use the GitHub option instead.
Left desired capacity at 2–3 over the weekendBurns budget for nothing and is itself a rubric line. Scale to 1 every time you step away.
Lost-state journal entry just says "it disappeared"Name why (in-memory storage isn't shared across instances) and which service fixes it (DynamoDB, coming in M4) — the explanation is what's graded, not the observation.

📱 Track B — You Already Have an App (Bring Your Own App, AWS)

🚦 Hard gate: everything below only counts with an approved proposal on file. Email your one-pager right after Class 3; if it isn't approved before Class 4, you submit under Track A (Pelican Tours) instead — no penalty, but decide that before Monday, not during class. Eligibility & proposal template: Eligibility & Setup page.

Same milestone, same 25 points, same deadline as Track A — the raw material is your own app instead of a fictional client. Full brief: Bring Your Own App project page.

📦 Exactly what you submit Monday

  • Your approved proposal (the email you sent + the instructor's approval) referenced or attached in your journal — this is the grading contract for your whole semester.
  • Private repo link with a course copy of your app on its own branch/fork (never your live version), plus JOURNAL.md and a screenshots/ folder with every screenshot used as evidence.
  • Component inventory table — every piece of your app (client, server, database, media/files, third-party services) mapped to the cloud service model it uses (IaaS / PaaS / SaaS / BaaS).
  • A live cloud URL — your backend running on EC2 behind an ALB, reachable right now.
  • Sizing justification (written) — measurements or reasoned estimates defending t3.micro (or your chosen size) for your workload.
  • Screenshot: load distributing across 2–3 instances of your backend behind the ALB (add an instance-id/hostname field to a health endpoint if you don't already have one).
  • Broken-state journal entry — what state (sessions, uploads, caches, in-memory data) broke when your backend scaled to multiple instances, specific to your app.
  • Second-device screenshot — your app's actual client (phone, emulator, or another browser) talking to the cloud-hosted backend, plus one sentence on what had to change (URLs, ports, CORS).
  • Metering baseline — requests served, CloudWatch resource metrics + a budget reading, and a projected monthly cost at your current traffic on a real provider.
  • Build-vs-buy write-up — one component of your app (auth, media hosting, database, push notifications, or payments) appraised keep-vs-replace with a managed/SaaS alternative, with a decision and reasons.
  • Proof the budget is respected — Auto Scaling desired capacity back down before you submit.
